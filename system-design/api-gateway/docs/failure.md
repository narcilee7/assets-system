# Failure Mode

## F1: 注册中心故障（服务发现失效）

### 场景

| 场景 | 影响 | 恢复方式 |
|------|------|----------|
| Consul 集群不可用 | 新实例无法注册，旧实例状态停滞 | 网关保持本地最后快照，允许手动降级 |
| etcd 分片故障 | 部分服务无法发现 | 一致性 Hash 将请求调度到健康分片 |
| 注册中心网络抖动 | 实例状态短暂不一致 | 依赖心跳超时保护（通常 10-30s） |

### 影响范围

```
Consul 故障
  │
  ├── 新服务上线 → 网关不知道 → 请求失败（503）
  │    └── 恢复：注册中心恢复后重新注册，Consul 补发历史事件
  │
  └── 旧服务下线 → 网关不知道 → 持续往已下线实例发请求
       └── 恢复：健康检查探测到实例不可用，从 index 移除
```

### 应对策略

1. **本地快照兜底**：每个网关节点在本地缓存一份实例列表，Redis/etcd 不可用时用本地快照
2. **主动探测**：即使注册中心说实例 healthy，也要探测（端口检测 + HTTP 健康检查）
3. **降级策略**：注册中心完全不可用时，切换为**静态配置模式**（只路由到固定地址）
4. **告警**：注册中心失联超过 30s 触发 PagerDuty

## F2: Redis 不可用（限流/会话/熔断状态丢失）

### 场景

| 场景 | 影响 | 恢复方式 |
|------|------|----------|
| Redis 单节点故障 | 部分限流失效，Token 黑名单失效 | 切换到 Redis Cluster 主节点 |
| Redis Cluster 部分分片故障 | 部分 key 读写失败 | 客户端自动重定向到其他分片 |
| Redis 连接池耗尽 | 限流请求阻塞，影响网关自身 | fail-open（本地限流兜底） |

### 限流降级路径

```
请求进入限流判断
  │
  ├── L1 本地限流（In-Memory Token Bucket）
  │    命中且未超限 → 放行（95% 路径，零 Redis 调用）
  │
  ├── 本地将超限 → 查询 Redis
  │    │
  │    ├── Redis 可用 → 按 Redis 结果判定
  │    └── Redis 不可用
  │         │
  │         ├── 配置 fail-open → 放行（允许临时超限）
  │         └── 配置 fail-close → 拒绝（保守策略）
```

### 本地限流兜底实现

```go
type LocalRateLimiter struct {
    // 每个 route+dimension 一个本地桶
    buckets sync.Map  // map[string]*TokenBucket

    // 配置
    capacity  int64
    refillRate int64  // per second
}

func (l *LocalRateLimiter) Allow(key string) bool {
    bucket, _ := l.buckets.LoadOrStore(key, newTokenBucket(l.capacity, l.refillRate))
    return bucket.Allow(1)
}
```

**重要**：本地限流无法跨节点协同，但可以保护单个节点不被流量打垮。

## F3: 后端服务故障（502/504）

### 502 Bad Gateway（后端不可达）

| 原因 | 诊断 | 应对 |
|------|------|------|
| 后端服务进程崩溃 | 端口不可连接 | 运维重启，自动恢复 |
| 网络分区 | telnet 后端端口失败 | VPC 层面的网络告警 |
| 后端安全组阻止 | 从网关能访问，但后端拒绝连接 | 调整安全组规则 |
| 后端连接池耗尽 | 连接超时 | 扩大后端连接池 |

### 504 Gateway Timeout（后端响应超时）

| 原因 | 诊断 | 应对 |
|------|------|------|
| 后端处理慢 | 正常情况下 P99 500ms，突然变成 5s | 后端性能问题，需加缓存/优化 |
| 数据库慢查询 | 后端日志显示大量慢查询 | 优化数据库索引 |
| 后端 Full GC | GC 停顿导致请求堆积 | JVM 参数调优 |

### 502/504 自动恢复流程

```
后端返回 502/504
  │
  ▼
熔断器记录失败（error_count++）
  │
  ▼
错误率达到阈值（50%，窗口 10s）
  │
  ▼
熔断器 OPEN → 该服务所有请求快速失败（不经过后端）
  │
  ▼
30s 后进入 HALF_OPEN → 放行 1 个探测请求
  │
  ├── 探测成功 → CLOSED（逐步恢复流量）
  └── 探测失败 → OPEN（继续熔断 30s）
```

## F4: 网关自身故障（单点可用性）

### 核心问题

网关是所有流量的单入口，一旦网关崩溃，全站不可用。

### 高可用部署架构

```
                    ┌─────────────────────────┐
                    │     Load Balancer        │
                    │  （Tengine / HAProxy）   │
                    └──────────┬──────────────┘
                               │ VIP / L4
           ┌───────────────────┼───────────────────┐
           ▼                   ▼                   ▼
    ┌─────────────┐      ┌─────────────┐      ┌─────────────┐
    │  GW Node 1  │      │  GW Node 2  │      │  GW Node 3  │
    │  (active)   │      │  (active)   │      │  (active)   │
    └──────┬──────┘      └──────┬──────┘      └──────┬──────┘
           │                    │                    │
           └────────────────────┼────────────────────┘
                               ▼
                    ┌─────────────────────┐
                    │   Redis Cluster     │  ← 限流状态、Session
                    └─────────────────────┘

                    ┌─────────────────────┐
                    │   Config Center      │  ← 路由配置、规则
                    │  (etcd / Consul)    │
                    └─────────────────────┘
```

### 故障检测与自动切换

```
健康检查（每 5s）
  │
  ├── GW Node 1 健康 → 接收流量
  ├── GW Node 1 不健康（连续 3 次超时）
  │    ├── Load Balancer 将 Node 1 从 UP 列表移除
  │    ├── 流量自动调度到 Node 2 / Node 3
  │    └── 触发告警：网关节点故障
  │
  ▼
运维介入
  ├── 重启故障节点
  └── 自动加入集群（配置中心 watch 自动同步路由表）
```

## F5: 路由规则冲突与异常匹配

### 路由优先级歧义

| 问题 | 场景 | 影响 |
|------|------|------|
| 双通配符冲突 | `/api/**` 和 `/api/v2/**` | 不确定性匹配 |
| 动态路径过长 | `/users/{id}` 和 `/users/me` | `{id}` 匹配到 "me" |
| 正则回溯灾难 | `/api/.*/(user\|admin)` 大量嵌套 | CPU 100%，请求超时 |

### 路由匹配优先级规则

```
1. 优先级数值越小越优先（显式设置）
2. 精确路径优先于前缀路径
3. 前缀路径按深度优先（更长的路径优先）
4. 正则匹配在最末位兜底
```

### 正则回溯灾难（ReDoS）防护

```go
// 路由规则正则必须在入库前通过 ReDoS 检测
func validateRegex(pattern string) error {
    regex := regexp.MustCompile(pattern)
    testInput := strings.Repeat("a", 30) + "b"  // 构造可能触发回溯的输入

    done := make(chan error, 1)
    go func() {
        regex.MatchString(testInput)
        done <- nil
    }()

    select {
    case <-done:
        return nil  // 通过
    case <-time.After(100 * time.Millisecond):
        return fmt.Errorf("regex %q 可能触发回溯灾难", pattern)  // 拒绝入库
    }
}
```

## F6: JWT 验证失败的各种场景

### JWKS 缓存击穿

| 场景 | 影响 | 应对 |
|------|------|------|
| JWKS 过期，刚好有新 Kid 请求 | 所有 JWT 验证失败 | 异步刷新，使用旧缓存兜底 |
| Auth 服务完全不可用 | JWKS 刷新失败，使用过期缓存 | 缓存过期时间设上限（最大 24h） |
| Token 算法切换（RS256→ES256） | 旧 Kid 对应新算法无法验证 | 渐进切换，留足迁移窗口 |

### Token 验证失败降级

```
JWT 验证失败
  │
  ├── 签名验证失败（Kid 不存在）
  │    ├── 触发 JWKS 异步刷新
  │    └── 当前请求使用旧缓存尝试验证（RS256→RS256 才可能）
  │
  ├── Token 过期
  │    └── 返回 401 + WWW-Authenticate Header
  │
  ├── Token 被撤销（jti 在黑名单）
  │    └── 返回 401（安全场景）
  │
  └── 缺少必要 Claim
       └── 返回 403（权限不足）
```

## F7: 熔断器误判（false positive）

### 场景

| 场景 | 结果 | 应对 |
|------|------|------|
| 后端 GC 停顿（5-10s） | 误判为后端故障，触发熔断 | 扩大统计窗口，使用滚动窗口而非固定窗口 |
| 依赖的 DB 慢查询（单次） | 单次超时触发错误率超标 | 设置错误率阈值和最小请求数（避免低 QVU 误判） |
| 健康检查本身失败 | 健康检查错误导致实例被移除 | 分离健康检查路径和业务路径 |
| 网络抖动（< 500ms） | 偶发 502 累积触发熔断 | 设置熔断的最短持续时间（避免频繁状态切换） |

### 滚动窗口统计

```go
type SlidingWindowCounter struct {
    windowSize  int64      // 窗口大小（秒）
    bucketSize  int64      // 子桶大小（秒）
    buckets     []*RingBuffer
    mu          sync.Mutex
    currentIdx  int64
}

func (c *SlidingWindowCounter) Record(result RequestResult) {
    idx := c.currentIdx % c.numBuckets
    c.buckets[idx].Add(result)
}
```

## F8: 配置热更新引发的一致性问题

### 场景

路由规则热更新时，新旧请求使用了不同的路由规则，可能导致：
- 同一请求先路由到 A 服务，再路由到 B 服务（不一致）
- 限流规则变更导致已通过的请求被重复限流

### 应对策略

```
版本化的路由配置
  │
  ├── 每个配置变更产生递增的 version number
  ├── 配置推送时带上目标版本号
  │
  ├── 请求处理时记录当前使用的配置版本
  │    └── 日志中记录 route_version="v42"
  │
  └── 如果版本不匹配，触发配置刷新后重试
       └── 避免使用不一致的配置版本处理请求
```

## F9: 大文件上传导致的内存问题

| 场景 | 影响 | 应对 |
|------|------|------|
| 客户端上传 1GB 文件 | 网关内存被打爆 | 设置 body 大小限制（默认 100MB）；大文件走直连 storage |
| 分片上传时路由变更 | 路由到不同后端，文件不完整 | 分片上传期间锁定使用同一后端（session affinity） |
| WebSocket 升级后长连接 | 连接占用内存，长连接数受限于 ulimit | 设置连接超时、最大连接数限制 |

## F10: DDoS 攻击导致网关过载

### 场景

| 攻击类型 | 表现 | 应对 |
|----------|------|------|
| 大规模 CC 攻击 | 10W QPS 正常请求涌入 | 限流（入口）+ IP 信誉库联动 |
| Slow Loris | 大量慢连接占用连接池 | 读取超时限制（read_timeout < 30s） |
| 放大攻击 | 请求小，响应大（带宽放大） | 限制后端响应大小；缓存热门响应 |
| DNS 重新绑定 | 欺骗健康检查 | 健康检查使用固定 IP，不依赖 DNS |

### 流量清洗流程

```
DDoS 流量
  │
  ├── 运营商层清洗（如 Cloudflare）
  │    ├── 清洗层无法识别的攻击
  │    │
  │    ▼
  │  WAF 层（应用层攻击）
  │    │
  │    ▼
  │  网关层限流
  │    │
  │    ▼
  │  后端服务（最终保护）
  │
  └── 正常流量 → 到达后端
```
