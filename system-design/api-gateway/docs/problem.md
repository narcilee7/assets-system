# Problem

## 需求

构建一个生产级的 API 网关，作为所有外部流量的唯一入口，提供路由、认证、限流、熔断、协议转换和安全防护能力。

### 功能需求

#### 1. 路由与负载均衡

- **动态路由**：根据请求特征（路径、方法、Header、Query 参数）将请求转发到后端目标服务
- **服务发现**：支持从注册中心（ Consul / etcd / Nacos）自动感知后端实例变更
- **负载均衡**：支持轮询、加权、最小连接数、延迟感知等策略
- **路由热更新**：路由规则变更无需重启网关，近乎实时生效

#### 2. 认证与鉴权

- **多协议认证**：
  - API Key（简单场景）
  - Basic Auth（内部服务）
  - JWT / OAuth2（主流场景，支持 RS256/HS256）
  - mTLS（高安全要求场景）
- **集中鉴权**：网关统一拦截验证，而非每个服务重复实现
- **RBAC 权限模型**：基于角色的访问控制，支持路径级、服务级、操作级权限
- **Token 验证策略**：支持 JWKS 缓存、在线验证、Token 无状态化

#### 3. 限流

- **多粒度限流**：全局、用户、IP、接口、服务、组合维度
- **多算法**：令牌桶（平滑）、漏桶（严格速率）、滑动窗口（精度）
- **分布式限流**：基于 Redis，支持跨节点协同
- **独立配额**：VIP / 白名单独立配额，不受普通限流影响
- **限流范围**：入口限流（保护网关本身）和出口限流（保护后端服务）分离

#### 4. 熔断与降级

- **熔断器**：基于错误率、延迟、P99 指标自动触发熔断
- **健康检查**：主动探测后端服务可用性（HTTP / TCP / UDP 健康检查）
- **降级策略**：熔断后返回降级内容、调用备用服务、返回历史缓存
- **半开恢复**：熔断后定时放行探测请求，逐步恢复流量

#### 5. 协议转换

- **HTTP ↔ gRPC**：HTTP-JSON 到 gRPC-PROTOBUF 的协议转换
- **HTTP ↔ WebSocket**：协议升级和长连接管理
- **gRPC ↔ gRPC-Web**：浏览器端 gRPC 调用支持
- **SOAP ↔ REST**：企业遗留系统的兼容

#### 6. 安全防护

- **DDoS 防护**：基于速率的自动清洗，IP 信誉库联动
- **WAF 集成**：SQL 注入、XSS、路径遍历等攻击检测
- **黑白名单**：IP 黑白名单、地理区域限流
- **CORS**：跨域资源共享配置
- **Request ID 透传**：为每个请求生成唯一 ID，贯穿整个调用链

#### 7. 可观测性

- **全量日志**：请求级别日志（路径、延迟、状态码、来源）
- **指标采集**：QPS、延迟分布（P50/P90/P99/P999）、错误率、限流触发率
- **分布式追踪**：Header 透传（Trace ID / Span ID），支持 Jaeger/Zipkin
- **告警**：限流触发、熔断激活、后端服务异常、网关自身 CPU/内存异常

#### 8. 插件化架构

- **中间件链**：认证、鉴权、限流、熔断、日志等以插件形式串联
- **插件生命周期**：Load → Init → Setup → Request/Response → Cleanup
- **插件隔离**：插件间无共享状态，运行时隔离
- **自定义插件**：支持 WASM / Lua 编写自定义插件

---

### 非功能需求

| 指标 | 目标 | 说明 |
|------|------|------|
| 网关延迟 | P99 < 5ms | 不应成为系统瓶颈 |
| 吞吐量 | 单机 10W+ QPS | 纯网关逻辑 |
| 可用性 | 99.99% | 网关是单点，必须高可用 |
| 路由匹配性能 | < 100μs | 万级路由规则下 |
| 规则热更新延迟 | < 1s | 规则变更到全网生效 |
| 最大路由规则数 | 10W+ | 支持多租户、多业务线 |

---

### 约束

- 网关自身**不能有状态**：所有状态（路由表、限流计数）必须外置到 Redis/配置中心
- **失败策略**：网关故障时应 fail-open（放行流量），避免成为全站不可用的根因
- **协议假设**：默认处理 HTTP/1.1，HTTP/2 和 gRPC 需专项优化
- **多租户隔离**：不同租户的路由、限流、认证规则必须严格隔离

### 关键设计决策

#### 决策 1：网关放行还是拦截？

| 方案 | 描述 | 适用场景 |
|------|------|----------|
| **fail-open** | 网关故障时默认放行所有请求 | 非核心服务、追求可用性 |
| **fail-close** | 网关故障时默认拒绝所有请求 | 核心服务、严格安全要求 |
| **混合策略** | 根据接口重要性动态选择 | 生产环境推荐 |

**推荐**：默认 fail-open，关键接口（计费、鉴权）fail-close，通过配置控制。

#### 决策 2：认证在网关层还是服务层？

| 方案 | 网关层 | 服务层 |
|------|--------|--------|
| 认证逻辑 | 统一实现，集中管理 | 各服务重复实现 |
| 性能开销 | 每次请求多一次验证 | 无额外开销 |
| 灵活性 | 受网关能力限制 | 可深度定制 |
| 安全性 | 网关被攻破则全部暴露 | 服务可独立加固 |
| 场景 | 标准协议（JWT/OAuth2） | 企业特有认证协议 |

**推荐**：标准认证在网关层做，服务层做兜底校验（双重防护）。

#### 决策 3：路由如何匹配？

| 方案 | 实现 | 性能 | 灵活性 |
|------|------|------|--------|
| **精确匹配** | HashMap | O(1) | 低（仅精确路径） |
| **前缀匹配** | Radix Tree | O(k) | 中（路径前缀） |
| **正则匹配** | RE2 自动机 | O(n) | 高（复杂规则） |
| **混合匹配** | 分层索引 + Trie | O(k) | 高 |

**推荐**：分层匹配——精确匹配走 HashMap，前缀匹配走 Radix Tree，正则匹配走 RE2。

#### 决策 4：插件架构选型

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Lua** | 热更新、门槛低、性能好 | 生态弱、调试困难 |
| **WASM** | 沙箱隔离、多语言支持 | 冷启动开销、冷门生态 |
| **Go Plugin** | 原生性能、无额外开销 | 需重建、API 版本耦合 |
| **微内核 +  RPC** | 独立进程、热升级 | 延迟高、部署复杂 |

**推荐**：Lua + WASM 双轨——简单逻辑用 Lua，复杂/安全隔离逻辑用 WASM。

### 真实案例与踩坑

#### 案例 1：Netflix Zuul 的路由规则膨胀

Netflix 早期 Zuul 1 基于线程池模型，当路由规则超过 10K 时，正则匹配导致 CPU 飙高，请求队列积压，最终引发全站故障。

**教训**：路由规则量级上来后，正则匹配必须替换为 Radix Tree + 精确匹配的分层策略。

#### 案例 2：Kong 网关的限流精度陷阱

Kong 早期使用固定窗口限流，窗口边界时大量请求同时被接受/拒绝，导致后端被打爆。切换到滑动窗口后才解决。

**教训**：限流算法选型要考虑边界突发问题。

#### 案例 3：AWS API Gateway 的 29s 超时坑

AWS API Gateway 默认 29s 超时（与 ALB 一致），但很多客户端设置了更短的超时，导致大量连接在网关处被意外断开。

**教训**：网关层超时配置要与客户端、后端服务三方协同，不能单独决定。
