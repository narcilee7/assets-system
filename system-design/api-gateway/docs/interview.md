### 1. 网关如何做到高可用？

#### 面试官想知道

- 你是否理解网关作为流量单入口的风险
- 你是否有完整的 HA 设计思路
- 你是否理解故障检测和自动恢复机制

#### 讨论要点

```
高可用架构设计：

1. 冗余部署
   - 多节点并行（至少 3 节点）
   - 节点间无共享状态（所有状态外置）
   - Load Balancer 层健康检查（主动探测 + 自动摘除）

2. 故障检测
   - 节点级：Load Balancer 主动探测（TCP/HTTP 健康检查）
   - 服务级：健康检查独立路径，不走业务逻辑
   - 注册中心：实例心跳超时（默认 10s）自动标记 unhealthy

3. 快速恢复
   - 配置中心推送 + 本地缓存（路由配置不依赖节点本地存储）
   - 限流状态外置 Redis（节点重启不影响限流）
   - 插件状态无持久化（插件重新加载即可）

4. 灾难恢复
   - 节点级故障：Load Balancer 自动摘除，流量转移到健康节点
   - AZ 级故障：跨 AZ 部署，自动 failover
   - 全站故障：限流 fail-open，确保核心业务不因网关故障全停
```

#### 追问

**Q：如果 3 个节点中有 1 个故障，Load Balancer 需要多长时间发现？**

A：
- 健康检查间隔通常 5s
- 连续 3 次探测失败才摘除 = 15s 内发现并摘除
- 流量在新请求进来时自动转移到健康节点（已有连接保持）
- 总体恢复时间 < 20s

### 2. 熔断器如何设计？何时打开，何时恢复？

#### 面试官想知道

- 你是否理解熔断器状态机（closed/open/half_open）
- 你是否理解错误率和延迟的权衡
- 你是否能设计防误判的机制

#### 讨论要点

```
熔断器设计：

状态机：
  CLOSED →（错误率 > 50% 或 P99 > 2s）→ OPEN
  OPEN →（30s 后）→ HALF_OPEN
  HALF_OPEN →（连续 3 次探测成功）→ CLOSED
  HALF_OPEN →（探测失败）→ OPEN（再等 30s）

关键参数：
  - ErrorThreshold：50%（可调，太低易误判，太高不起作用）
  - LatencyThreshold：P99 > 2s（防后端慢性）
  - WindowSize：10s（统计窗口，太短易波动，太长反应慢）
  - RecoveryTimeout：30s（熔断持续时间，太短易震荡，太长延迟恢复）
  - MinRequestCount：100（最小请求数，低 QVU 下不触发熔断）

防误判：
  - 滚动窗口替代固定窗口（避免窗口边界效应）
  - 慢请求单独统计（P99 而非平均延迟）
  - 半开探测时逐步放量（1% → 10% → 50% → 100%）
```

#### 追问

**Q：如果熔断器频繁在 open/half_open/closed 之间震荡怎么办？**

A：
- 原因：后端服务不稳定，探测请求刚好成功但业务请求还是失败
- 解决：
  1. **增大熔断窗口**：30s → 60s → 120s（指数退避）
  2. **提高半开成功阈值**：连续 3 次成功 → 连续 5 次成功
  3. **分离统计维度**：健康检查用独立 HTTP 路径，不依赖业务路径
  4. **设置熔断最短持续时间**：OPEN 状态至少保持 60s

### 3. 如何保证路由规则的一致性？

#### 面试官想知道

- 你是否理解分布式配置的一致性问题
- 你是否理解配置中心的一致性模型（etcd/consul 的 Raft）

#### 讨论要点

```
配置一致性：

方案 1：配置中心推送（etcd / Consul Watch）
  - 配置变更 → 配置中心 Raft 复制 → Watch 触发推送
  - 优点：实时性强（< 1s）
  - 缺点：配置中心成为单点（但配置中心本身是高可用的 Raft 集群）

方案 2：定期拉取（每 5s）
  - 各节点定时拉取最新配置
  - 优点：简单，无推送依赖
  - 缺点：最长可能有 5s 延迟不一致

方案 3：混合（推送 + 拉取兜底）
  - 配置中心 Watch 推送（实时）
  - 定时器每 30s 强制拉取（兜底，防止推送丢失）
  - 结果：实时性 < 1s，同时有兜底

版本化配置：
  - 每个配置带 version number
  - 配置变更递增版本
  - 请求处理时验证配置版本，不一致的配置不用于处理请求
```

### 4. 限流算法的选择（令牌桶 vs 滑动窗口 vs 漏桶）

#### 面试官想知道

- 你是否理解各算法的原理和适用场景
- 你是否能根据业务场景选择合适的算法

#### 讨论要点

```
四种算法对比：

1. 令牌桶（Token Bucket）
   原理：桶里有令牌，请求需要拿令牌，无令牌则拒绝
   特点：允许突发流量（只要桶未满），平滑限流
   适用：大多数 API 限流场景（保护后端不过载但允许峰值）
   缺点：实现稍复杂（需要定时补充令牌）

2. 漏桶（Leaky Bucket）
   原理：请求入桶，桶以固定速率漏出，桶满则拒绝
   特点：严格平滑输出（以恒定速率处理请求）
   适用：下游带宽有限、需要严格限速的场景（如第三方 API 调用）
   缺点：不允许任何突发，新请求必须等旧请求处理完

3. 滑动窗口（Sliding Window）
   原理：将时间窗口分成 N 份，计算当前时刻往前 N 份内的请求数
   特点：比固定窗口平滑，避免边界突发
   适用：需要精确限流、不允许任何突发的场景（如支付接口）
   缺点：Redis 实现复杂（需要维护 N 个子窗口的计数）

4. 固定窗口（Fixed Window）
   原理：以固定时间窗口为周期，窗口内计数超限则拒绝
   特点：实现最简单，但窗口边界有突发问题
   适用：简单场景，临时方案
   缺点：例如每秒 100 次限制，在 1.9s 和 2.1s 各来 100 次 = 实际 200 次/秒

分布式场景下的算法选择：
  - 单节点：令牌桶（本地实现，无网络开销）
  - 多节点协同：Redis + Lua 脚本实现滑动窗口（精度最高）
  - 需要跨节点精确计数：Redis Cluster + 分片
```

#### 追问

**Q：如果 Redis 不可用，你的限流如何降级？**

A：
```
降级策略（按保守程度）：

1. Fail-open（默认）
   - 本地限流兜底（单节点令牌桶）
   - 优点：不影响业务，请求都能处理
   - 缺点：总 QPS 可能超过后端承受能力

2. Fail-close（保守）
   - 本地限流且限制更严格（单机配额 = 原始配额 / 节点数）
   - 优点：严格保护后端
   - 缺点：可能误杀正常请求

3. 降级到静态配置
   - 根据接口重要性分级：
     - 核心接口（计费）：fail-close
     - 一般接口：fail-open + 本地限流
     - 内部接口：无限制
```

### 5. JWT 验证的性能如何优化？

#### 面试官想知道

- 你是否理解 JWT 验证的性能瓶颈（CPU + 网络）
- 你是否理解 JWKS 缓存的重要性

#### 讨论要点

```
JWT 验证性能优化：

1. JWKS 缓存（最重要）
   - 每次验证都从 Auth 服务获取 JWKS = 10W QPS × JWKS 大小 = 网络爆炸
   - 解决方案：内存缓存 JWKS（TTL = min(kid exp, 1h)）
   - Kid 不存在时触发异步刷新（不阻塞当前请求）

2. 本地缓存验证结果
   - Token 有过期时间，过期内验证结果可复用
   - 缓存 key = SHA256(token 前 200 字符)
   - TTL = min(token exp - now, 60s)

3. 算法选择（按场景）
   - HS256：签名验证快 10x，但密钥需要安全存储
   - RS256：支持 Key 轮换，适合多服务场景
   - ES256：更短签名，但验签稍慢

4. 并发验证优化
   - 验证本身是 CPU 密集，Go 中用 goroutine 并发验证
   - 限制并发数（避免内存爆炸）
```

### 6. 如何设计插件化架构？

#### 面试官想知道

- 你是否理解插件化架构的设计模式
- 你是否理解插件隔离的重要性

#### 讨论要点

```
插件化架构设计：

1. 插件生命周期
   Load → Init → Setup → Request → Response → Cleanup

2. 插件链模型
   预编译插件链（启动时确定顺序）
   请求进入 → Plugin[0] → Plugin[1] → ... → Plugin[N] → 后端
                              ↑
                         如果 Aborted，跳过剩余插件

3. 插件隔离
   - 插件间无共享状态（通过 PluginContext 传递数据）
   - 插件 panic 不影响网关主流程（defer recover）
   - WASM 插件有独立沙箱（无原生 Go runtime）

4. 插件配置热更新
   - 插件配置存储在配置中心
   - 配置变更 → 触发插件重新 Setup
   - 不中断请求处理（只影响新请求）

5. 自定义插件扩展
   - Lua：轻量，嵌入式（Kong 风格）
   - WASM：安全隔离，多语言支持（Solo.io Envoy 风格）
   - RPC Plugin：独立进程，语言无关（gRPC Plugin）
```

### 7. 网关的安全设计（零信任）

#### 面试官想知道

- 你是否理解零信任安全模型
- 你是否理解网关层的安全防护职责

#### 讨论要点

```
零信任网关安全：

1. 认证（Authentication）
   - 强身份验证：每次请求都必须认证，无"可信内网"概念
   - 支持多种协议：JWT/OAuth2/API Key/mTLS
   - Token 无状态化：验证时不查询数据库

2. 鉴权（Authorization）
   - RBAC：基于角色，不是基于用户
   - ABAC：基于属性（租户、时间段、IP）动态决策
   - 最小权限：每个 Token 只授权必要的路径

3. 传输安全
   - TLS 1.3（现代加密，零 RTT）
   - mTLS（服务间双向认证）
   - 证书自动轮换

4. 防护
   - DDoS：速率限制 + IP 信誉库
   - WAF：SQL 注入、XSS、路径遍历
   - 审计：所有请求的完整日志（特别是敏感操作）

5. 密钥管理
   - 网关不存储密钥，密钥在 Vault 中
   - 动态密钥轮换
   - Secret Scanning：日志中不能出现明文密钥
```

### 8. 生产环境踩过的坑

#### 面试官想知道

- 你是否有真实的故障处理经验
- 你是否能从故障中总结系统性改进

#### 典型案例

```
案例 1：正则匹配导致 CPU 100%

现象：网关 P99 延迟从 5ms 突然飙到 500ms，CPU 100%
根因：产品同学配置了一条回溯灾难的正则（/api/.*.*.*.*user/.*/admin/）
     当请求路径较长时，RE2 引擎进入指数级回溯
解决：
  1. 上线前用 ReDoS 检测工具扫描所有正则规则
  2. 限制正则数量，正则路由占比 < 1%
  3. 增加正则匹配超时（100ms 超时强制终止）
教训：正则是网关的性能陷阱，要严格控制

---

案例 2：JWT 黑名单导致 Redis 崩溃

现象：Token 撤销（退出登录）后，Redis 写入量暴增 10x
根因：每次请求都要检查 Token JTI 是否在黑名单
     10W QPS × Redis 查询 = Redis CPU 打满
解决：
  1. 布隆过滤器（Bloom Filter）做快速判断（可能有 false positive 但不能有 false negative）
  2. 布隆过滤器 Miss 时才查 Redis
  3. 结果：Redis 查询量降低 90%，CPU 恢复正常
教训：高频路径上的每一次额外查询都要考虑级联影响

---

案例 3：网关节点内存泄漏

现象：网关运行 3 天后内存从 500MB 涨到 8GB，开始 OOM
根因：Radix Trie 节点对象没有正确 GC，
     每更新一次路由规则都创建新节点，旧节点没释放
解决：
  1. 使用只读 Radix Trie（不可变，更新时重建整棵树）
  2. 每次更新替换根节点指针（原子操作）
教训：网关是长期运行进程，任何累积性的资源泄漏都要在测试阶段发现
```
