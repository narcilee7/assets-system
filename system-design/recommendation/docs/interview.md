### 1. 推荐系统的漏斗是如何设计的？

#### 面试官想知道

- 你是否理解推荐系统的分层漏斗
- 你是否能根据业务场景设计漏斗参数
- 你是否理解每一层的计算量和延迟约束

#### 讨论要点

```
推荐漏斗设计：

阶段 1：召回（Recall）
  - 输入：全量物品（千万~亿级）
  - 输出：候选集（千~万级）
  - 要求：快（< 20ms），覆盖广
  - 方法：多路召回（CF、Embedding、热度、标签）

阶段 2：粗排（Coarse Ranking）
  - 输入：万级候选
  - 输出：百~千级
  - 要求：中（< 15ms），轻量模型
  - 方法：轻量 GBDT、规则排序

阶段 3：精排（Fine Ranking）
  - 输入：百~千级
  - 输出：十~百级
  - 要求：准（< 30ms），复杂模型
  - 方法：DNN、DeepFM、DIN

阶段 4：重排（Re-ranking）
  - 输入：百级
  - 输出：十~二十级
  - 要求：多样化（< 10ms）
  - 方法：DPP、MMR、时间衰减

漏斗参数选择：
  - 召回量：取决于后端计算能力，通常 5000-10000
  - 粗排量：通常 500-1000
  - 精排量：通常 50-100
  - 返回量：通常 10-20
```

#### 追问

**Q：如果要求 P99 < 50ms，整个漏斗如何设计？**

A：
```
约束：总延迟 < 50ms

优化方案：
  1. 召回层并行计算（5ms）
  2. 粗排精简（5ms）
  3. 精排模型蒸馏，小模型（10ms）
  4. 特征预取（下一次请求提前拉取）
  5. 结果缓存（热点用户 30s 缓存）

或者：
  - 减少精排候选量（从 100 → 30）
  - 用规则排序替代部分精排
  - 增加在线计算资源
```

### 2. 如何解决冷启动问题？

#### 面试官想知道

- 你是否理解冷启动的多种场景
- 你是否有系统的解决方案
- 你是否能根据业务特点选择方案

#### 讨论要点

```
冷启动场景：

1. 用户冷启动
   方案：
     - 基于人口属性推荐（注册时收集）
     - 热门推荐兜底
     - 注册问卷快速获取兴趣
     - 跨平台数据导入

2. 物品冷启动
   方案：
     - 冷启动保序曝光（强制曝光 N 次）
     - 作者权重加成（优质作者优先）
     - 内容分析（NLP 提取标签）
     - Bandit 探索（以一定概率推荐新内容）

3. 系统冷启动
   方案：
     - 专家规则
     - 协同过滤 + 逐步学习
     - 公开数据集预训练

关键指标监控：
  - 冷启动物品的 CTR（验证效果）
  - 冷启动物品的曝光覆盖率
  - 冷启动用户的后续留存
```

#### 追问

**Q：如何量化冷启动的效果？**

A：
```
量化指标：

1. 冷启动物品覆盖率
   = 冷启动物品曝光数 / 冷启动物品总数
   - 理想值 > 50%（过半物品有曝光）

2. 冷启动 CTR
   = 冷启动物品点击数 / 冷启动物品曝光数
   - 和整体 CTR 对比，验证推荐质量

3. 冷启动用户留存
   = 冷启动用户 7 天后仍活跃的比例
   - 和正常用户留存对比

4. 长期效果
   - 冷启动物品最终的转化率和长期留存
   - 冷启动扶持应该逐步衰减，让自然竞争决定
```

### 3. 多路召回如何融合？

#### 面试官想知道

- 你是否理解多路召回的目的
- 你是否能设计合理的融合策略
- 你是否理解不同召回通道的特点

#### 讨论要点

```
多路召回融合策略：

融合方法：

1. 分数归一化后加权求和
   - 每路召回分数归一化到 [0, 1]
   - 按权重加权求和
   - 权重根据业务目标调整

2. 分数幂变换
   - score = w1 × score1^p1 + w2 × score2^p2 + ...
   - 幂次越高，该通道影响越大

3. 交叉融合
   - 先各路独立召回
   - 中间结果做交叉过滤（如协同过滤 + 标签匹配）
   - 最后合并

4. 序列融合
   - 先召回路 A → 结果作为路 B 的候选
   - 按序列依次融合

权重设置：
  - 协同过滤：0.3（个性化强）
  - Embedding：0.4（综合效果最好）
  - 热度：0.2（保底）
  - 标签：0.1（辅助）
```

#### 追问

**Q：如何发现和调整召回通道的权重？**

A：
```
方法：

1. 离线评估
   - 各通道单独评估 AUC/Recall
   - 计算通道贡献度（Shapley Value）

2. 在线 AB 测试
   - 调整权重比例
   - 观察 CTR、覆盖率、多样性变化

3. 自动调参
   - 使用 Bandit 算法动态调整
   - Exploration + Exploitation

监控指标：
  - 各通道召回量（是否均衡）
  - 各通道召回物品的平均分数
  - 各通道召回物品的去重率
```

### 4. 如何平衡准确性和多样性？

#### 面试官想知道

- 你是否理解推荐系统的核心矛盾（准确性 vs 多样性）
- 你是否有系统的解决方案
- 你是否能量化多样性

#### 讨论要点

```
准确性 vs 多样性：

问题描述：
  - 只追求准确性：推荐越来越集中，用户兴趣窄化
  - 只追求多样性：CTR 下降，用户体验差

量化指标：
  - Intra-List Diversity（ILD）：推荐列表内物品的分散度
  - 覆盖率：推荐物品占长尾物品的比例
  - 新颖性：用户未见过物品的比例

解决方案：

1. DPP 多样性重排
   - 在保证相关性的前提下，最大化多样性
   - 贪心选择边际效用最大的物品

2. MMR（Maximal Marginal Relevance）
   - relevance - λ × diversity
   - 平衡相关性和多样性

3. 约束规则
   - 同一作者最多出现 N 次
   - 同一标签不能超过 30%
   - 强制穿插不同类别

4. Exploration 机制
   - Epsilon-Greedy：一定概率随机推荐
   - UCB：不确定性高的优先推荐
```

#### 追问

**Q：如何验证多样性优化的效果？**

A：
```
验证方法：

1. 短期指标
   - ILD 多样性分数提升
   - 覆盖率提升
   - 新颖性提升

2. 长期指标
   - 用户留存提升
   - 人均消费时长提升
   - 用户兴趣广度（标签覆盖）提升

3. AB 测试
   - 对照组：只优化准确性
   - 实验组：准确性 + 多样性优化
   - 观察 2-4 周的长期效果

注意事项：
  - 多样性优化可能有短期 CTR 下降
  - 需要坚持观察 2-4 周再看长期效果
  - 避免过度优化（多样性过高也会伤害体验）
```

### 5. 推荐模型的实时更新如何实现？

#### 面试官想知道

- 你是否理解在线/离线架构的协同
- 你是否能设计特征和模型的更新流程
- 你是否理解数据漂移的风险

#### 讨论要点

```
实时更新架构：

实时层（分钟级）：
  - 用户短期兴趣：基于最近 5-30 分钟行为
  - Kafka + Flink 实时计算
  - 更新 Redis

近线层（小时级）：
  - 用户长期兴趣：小时级聚合
  - 物品统计量（CTR、完播率）：小时级更新
  - 增量训练

离线层（天级）：
  - 用户 Embedding：全量重训练
  - 物品 Embedding：全量重训练
  - 全量模型训练

同步机制：
  - 特征版本号：每次特征更新递增版本
  - 模型版本号：每次模型更新递增版本
  - 特征和模型必须对齐（不能用新特征 + 旧模型）
```

#### 追问

**Q：如何检测和处理数据漂移？**

A：
```
数据漂移检测：

1. 概念漂移
   - 监控 CTR 变化（显著变化说明漂移）
   - CUSUM 检测算法

2. 协变量漂移
   - 监控特征分布（PSI > 0.2 说明漂移）
   - 对比训练集和测试集特征分布

处理方法：

1. 模型重新训练
   - 检测到漂移后立即触发
   - 增量训练 + 全量校正

2. 在线学习
   - 实时更新模型参数
   - FTRL 等在线学习算法

3. 模型版本回滚
   - 如果新模型效果差，回滚到旧版本
   - 保留历史版本
```

### 6. 如何做推荐系统的 A/B 测试？

#### 面试官想知道

- 你是否理解 A/B 测试的设计原则
- 你是否能处理 A/B 测试中的坑
- 你是否理解流量分层和正交性

#### 讨论要点

```
A/B 测试设计：

1. 流量分层
   - 层之间正交（一个用户只在一个层）
   - 层内流量分流（对照组 + 实验组）

2. 分桶算法
   - hash(user_id + experiment_id + layer_id) % 100
   - 保证同一用户每次分到同一桶

3. 样本量计算
   - baseline_ctr = 4.5%
   - mde（最小可检测效果）= 5%
   - significance = 0.05
   - power = 0.8
   - 计算：每组需要 ~10 万样本

4. 实验周期
   - 通常 2-4 周
   - 覆盖完整周期（工作日 + 周末）
   - 前 3-5 天为 burn-in 期

5. 统计检验
   - t-test 计算 p-value
   - 置信区间
   - 效果量和置信区间
```

#### 追问

**Q：AB 实验结果不显著怎么办？**

A：
```
情况分析：

1. 样本量不足
   - 延长实验周期
   - 扩大流量

2. 效果太小
   - 检查原始指标设计是否合理
   - 也许效果确实很小，不值得上线

3. 指标选择问题
   - 核心指标不显著，但副指标显著
   - 分析副指标变化原因

4. 新奇效应
   - 用户对新功能好奇，初期效果不真实
   - 等新奇效应消退后再评估

决策原则：
  - p > 0.05：不能上线
  - p < 0.05，但效果太小：评估 ROI
  - p < 0.05，效果显著：可以上线
```

### 7. 如何处理推荐系统中的偏见问题？

#### 面试官想知道

- 你是否理解推荐系统中的偏见来源
- 你是否有公平性意识
- 你是否能设计去偏策略

#### 讨论要点

```
偏见来源：

1. 选择偏见
   - 用户选择的数据不代表真实偏好
   - 用户只看推荐的内容，忽视其他内容

2. 曝光偏见
   - 模型过度推荐曝光多的物品
   - 热门物品越来越热门

3. 反馈偏见
   - 模型只学习被点击的内容
   - 未曝光物品永远没机会

4. 社会偏见
   - 模型学习到历史上的歧视模式
   - 对特定群体不公平

去偏策略：

1. 曝光控制
   - 给未曝光物品保序曝光机会
   - 平衡热门和长尾

2. 训练数据去偏
   - IPS（Inverse Propensity Score）加权
   - 热门物品降采样

3. 公平性约束
   - 在模型优化目标中加入公平性约束
   - 群体间的推荐质量差异不能过大

4. 后处理
   - 重新排序时考虑公平性
   - 对弱势群体内容加权
```

#### 追问

**Q：如何监控推荐公平性？**

A：
```
监控指标：

1. 群体覆盖率差异
   - 不同群体的物品曝光比例差异
   - 差异不能超过阈值

2. 群体 CTR 差异
   - 不同群体对推荐内容的满意度差异
   - 差异过大需要调查

3. 长尾覆盖率
   - 推荐的物品中，长尾物品占比
   - 覆盖率 < 30% 需要关注

4. 内容多样性
   - 推荐内容的类别/标签分散度
   - 过于集中说明有偏见

告警：
  - 群体覆盖率差异 > 20%
  - 特定群体满意度持续低于平均
```

### 8. 生产环境踩过的坑

#### 面试官想知道

- 你是否有真实的故障处理经验
- 你是否能从故障中总结系统性改进

#### 典型案例

```
案例 1：协同过滤导致的信息茧房

现象：某视频平台上线协同过滤后，用户播放历史越来越集中
       用户反映推荐内容越来越单一，留存下降

根因：
  - 协同过滤有强化效应，用户只看推荐的内容
  - 推荐越来越集中，信息茧房

解决：
  1. 上线多样性指标（ILD、覆盖率）
  2. 引入 DPP 重排，保证多样性
  3. 强制曝光未看过内容（Exploration）
  4. 定期注入随机性

教训：协同过滤必须配合多样性优化，不能单独使用

---

案例 2：Embedding 更新延迟导致的效果下降

现象：小时级增量训练模型，但 Embedding 天级更新
       推荐效果在 Embedding 更新那天有明显波动

根因：
  - 用户 Embedding 和物品 Embedding 不同步
  - 用户用新模型排序，但物品 Embedding 是旧的
  - 不匹配导致效果下降

解决：
  1. Embedding 和模型必须同步更新
  2. 设计在线/离线架构对齐机制
  3. 监控特征和模型的时效性差异

教训：在线/离线架构必须对齐，不能孤立优化

---

案例 3：AB 实验流量不均导致结论不置信

现象：新功能 AB 测试显示 CTR 提升 10%，但上线后无效

根因：
  - 分桶算法问题，实验组和对照组流量不均
  - 实验组恰巧分配了高活跃用户

解决：
  1. 修复分桶算法，确保均匀
  2. 验证实验组和对照组用户特征分布一致
  3. 延长观察期，等统计显著

教训：AB 测试前必须验证流量均匀，不能只看结果
```
