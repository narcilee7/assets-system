# Problem

## 需求

构建一个生产级推荐系统，从海量候选内容中为目标用户精准推荐个性化内容，同时平衡准确性、多样性、新颖性和实时性。

### 功能需求

#### 1. 多路召回（Multi-Channel Recall）

召回是推荐系统的第一道漏斗，从海量内容（千万~亿级）中快速筛选出候选集（百~千级）。

| 召回通道 | 原理 | 优点 | 缺点 |
|----------|------|------|------|
| **协同过滤（CF）** | 根据用户历史行为找到相似用户/物品 | 强个性化，发现潜在兴趣 | 冷启动问题，稀疏性 |
| **向量召回（Embedding）** | 用向量表示用户和物品，余弦相似度召回 | 可离线计算，支持异构内容 | 计算量大，解释性差 |
| **热度召回** | 召回近期热门内容 | 简单有效，新内容有曝光 | 缺乏个性化 |
| **基于标签召回** | 匹配用户标签和内容标签 | 可解释，实现简单 | 标签质量依赖 |
| **地域召回** | 基于用户当前位置召回 | 场景强相关 | 隐私问题 |
| **社交召回** | 召回好友互动/关注的内容 | 信任度高 | 隐私问题 |
| **冷召回** | 新内容保序曝光 | 解决冷启动 | 效果不确定 |

#### 2. 粗排（Coarse Ranking）

粗排是用轻量级模型对多路召回的结果进行初步排序，从百~千级筛到十~百级。

| 方法 | 说明 | 计算量 |
|------|------|--------|
| **规则排序** | 热度 × 质量分 × 效率 | 极低 |
| **ML 轻量模型** | 轻量级 GBDT/FM | 低 |
| **向量相似度** | 用 embedding 直接排序 | 中 |

#### 3. 精排（Fine Ranking）

精排是用复杂模型对候选集进行精细排序，是推荐系统的核心环节。

| 模型 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **DNN** | 深度神经网络学习非线性特征交互 | 效果好，表达能力强 | 可解释性差，训练慢 |
| **DeepFM** | 结合 FM 和 DNN | 兼顾低阶和高阶特征交互 | 计算量大 |
| **DCN** | 交叉网络学习特征交互 | 高效，显式交叉 | 参数量大 |
| **DIN** | 注意力机制建模用户兴趣 | 对候选商品加权 | 只适用于序列 |
| **M5** | 多任务学习 | 共享表达，节省资源 | 任务间可能冲突 |
| **Embedding + MLP** | 经典范式 | 简单易扩展 | 特征交互能力弱 |

#### 4. 重排（Re-ranking）

重排是在精排之后，综合多样性、新颖性、商业目标进行最终调整。

| 目标 | 方法 | 说明 |
|------|------|------|
| **多样性** | MMR、DPP | 保证推荐结果不过于集中 |
| **新颖性** | 过滤已曝光内容 | 推荐用户未见过的新内容 |
| **商业目标** | 插入广告、扶持新内容 | 业务需求 |
| **时效性** | 时间衰减因子 | 越新的内容排序越高 |
| **Exploration** | Epsilon-Greedy、UCB | 探索用户新兴趣 |

#### 5. 特征工程

| 特征类型 | 包含 | 更新频率 |
|----------|------|----------|
| **用户特征** | 人口属性、兴趣标签、行为统计 | 天级/实时 |
| **物品特征** | 类别、标签、质量分、统计量 | 天级/实时 |
| **交叉特征** | 用户×物品交互统计 | 实时 |
| **上下文特征** | 时间、位置、设备、场景 | 实时 |
| **序列特征** | 用户行为序列 last-N | 实时 |

#### 6. 冷启动（Cold Start）

| 类型 | 场景 | 解决方案 |
|------|------|----------|
| **用户冷启动** | 新用户无行为数据 | 人口属性推断 + 热门推荐 + 注册问卷 |
| **物品冷启动** | 新内容无曝光数据 | 冷召回、作者权重、新内容保序 |
| **系统冷启动** | 新系统无任何数据 | 专家规则 + 协同过滤 + 逐步学习 |

#### 7. 反馈闭环（Feedback Loop）

```
用户行为（曝光 → 点击 → 播放 → 完播 → 互动）
  │
  ▼
行为日志（Kafka）
  │
  ├── → 实时特征更新（用户短期兴趣）
  ├── → 离线数据仓库（训练样本）
  └── → 模型训练（天级/小时级）
        │
        ▼
新模型上线（AB 测试）
  │
  ▼
优化推荐效果（闭环完成）
```

### 非功能需求

| 指标 | 目标 | 说明 |
|------|------|------|
| 推荐延迟 P99 | < 100ms | 从请求到返回结果 |
| 推荐候选集规模 | 召回万级 → 精排百级 → 返回十级 | 漏斗逐层递减 |
| 推荐的准确率（CTR）| 提升 5-10% | AB 测试验证 |
| 推荐的覆盖率 | > 60% | 长尾内容曝光比例 |
| 推荐的多样性 | DIVERSITY score > 0.7 | 不明显集中 |
| 系统可用性 | 99.99% | 推荐服务本身 |
| 模型更新频率 | 至少天级，最佳小时级 | 保证实时性 |

---

### 约束

- **候选集规模**：从千万/亿级内容中推荐，毫秒级返回
- **特征实时性**：用户兴趣可能小时级变化，特征需要实时更新
- **推荐公平性**：不能只推热门，长尾内容需要曝光机会
- **用户隐私**：行为数据的使用需要合规
- **模型可解释性**：至少在内部可解释，用于优化和 debug
- **A/B 测试**：所有改动必须通过 A/B 测试验证

### 关键设计决策

#### 决策 1：在线计算 vs 离线计算

| 阶段 | 策略 | 适用场景 |
|------|------|----------|
| **召回** | 离线计算 + 在线缓存 | 协同过滤 Embedding、热度榜 |
| **粗排** | 离线计算 + 在线微调 | 候选集大，实时性要求不高 |
| **精排** | 在线实时计算 | 需要最新特征和模型 |
| **重排** | 在线实时计算 | 需综合实时上下文 |

**推荐**：**离线计算 Embedding，在线计算排序**（平衡效果和实时性）

#### 决策 2：模型更新频率

| 策略 | 更新频率 | 优点 | 缺点 |
|------|----------|------|------|
| **天级全量** | 每天全量训练 | 稳定，代码简单 | 实时性差 |
| **小时级增量** | 每小时增量训练 | 实时性较好 | 可能有漂移 |
| **实时增量** | 流式训练（Flink）| 实时性最好 | 实现复杂，漂移风险 |
| **模型蒸馏** | 大模型蒸馏到小模型 | 在线可用 | 有精度损失 |

**推荐**：**小时级增量 + 每日全量校正**（平衡实时性和稳定性）

#### 决策 3：特征存储架构

| 方案 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| **在线特征存储（Feature Store）** | Redis/内存存储实时特征 | 低延迟 | 增加架构复杂度 |
| **特征随请求传入** | 把特征从外部服务拉取 | 简单 | 增加链路延迟 |
| **嵌入模型** | 特征直接编码到 Embedding | 简单 | 解释性差 |

**推荐**：**Feature Store + 请求传入混合**

#### 决策 4：推荐结果的多样性 vs 准确性

```
准确性问题：
  - 只推荐用户已知的热门内容 → 点击率高但无新意
  - 用户兴趣越来越窄（信息茧房）

多样性问题：
  - 大量推荐随机内容 → 多样性好但点击率低

解决方案：DPP（Determinantal Point Process）
  - 在保证相关性的前提下，最大化多样性
  - 优化目标：max diversity subject to relevance threshold
```

### 真实案例与踩坑

#### 案例 1：协同过滤导致的信息茧房

某视频平台用协同过滤推荐，用户播放历史集中于某类型视频后，推荐结果越来越窄，最终用户流失。

**教训**：
- 协同过滤有强化效应，需要 Exploration 机制
- 要有 diversity 指标约束推荐结果
- 定期注入随机性，避免用户陷入信息茧房

#### 案例 2：Embedding 更新延迟导致的效果下降

模型是小时级更新，但 Embedding 是天级更新，导致用户Embedding 和内容 Embedding 时间差大，推荐效果下降。

**教训**：
- Embedding 和模型必须同步更新
- 在线/离线架构必须对齐更新时间
- 监控特征和模型的时效性指标

#### 案例 3：冷启动物品的流量倾斜

新内容上线后，协同过滤无法推荐，导致曝光为 0。为扶持新内容调整权重后，旧内容的 CTR 下降 15%。

**教训**：
- 冷启动扶持需要精细设计（时间窗口、曝光量上限）
- 扶持后期望自然衰减，不能无限扶持
- 监控冷启动物品的转化率和长期留存
