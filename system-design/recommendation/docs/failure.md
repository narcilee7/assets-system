# Failure Mode

## F1: 冷启动问题（Cold Start）

### 场景

| 类型 | 场景 | 影响 |
|------|------|------|
| **用户冷启动** | 新用户注册，无历史行为 | 推荐效果差，可能流失 |
| **物品冷启动** | 新内容上线，无曝光数据 | 曝光为 0，作者积极性下降 |
| **系统冷启动** | 新平台上线，无任何数据 | 无法推荐 |

### 应对策略

#### 用户冷启动

```
策略 1：基于人口属性推荐
  - 用户注册时收集年龄、性别、兴趣领域
  - 用人口统计学的统计规律推荐
  - 例：18-25 岁男性 → 推荐科技、游戏内容

策略 2：热门推荐兜底
  - 新用户默认推荐全局热门内容
  - 随用户行为积累，逐步个性化

策略 3：注册问卷
  - 初始兴趣选择（5-10 个标签）
  - 快速构建初始兴趣画像
  - 权衡：问卷太长用户流失

策略 4：跨平台数据导入
  - 第三方登录获取基础画像
  - 微信获取 openid，对应微信画像
```

#### 物品冷启动

```
策略 1：冷启动保序曝光
  - 新内容强制曝光 N 次（100-500 次）
  - 用小流量测试效果
  - 如果 CTR > 基准，继续放量

策略 2：作者权重加成
  - 优质作者的内容加权
  - KOL（关键意见领袖）保序

策略 3：内容分析推荐
  - 用 NLP 分析内容标题/描述
  - 提取标签/关键词
  - 匹配喜欢类似内容的用户

策略 4：Bandit 探索
  - Epsilon-Greedy / UCB
  - 以一定概率推荐新内容
  - 平衡 Exploration 和 Exploitation
```

## F2: 推荐结果集中（信息茧房）

### 场景

| 问题 | 表现 | 根因 |
|------|------|------|
| **内容集中** | 推荐结果都是同一类内容 | 协同过滤强化效应 |
| **用户兴趣窄化** | 用户只看某几类内容 | 模型学习到的点击反馈循环 |
| **长尾曝光不足** | 曝光集中于头部内容 | 热度推荐权重过高 |

### 应对策略

#### 1. 多样性指标监控

```go
type DiversityMetrics struct {
    // Intra-List Diversity（ILD）
    // 基于类别分散度
    // ILD = 1 - avg_category_overlap_in_top20
    ILD float64  // > 0.7 为好

    // 覆盖率
    // 每天曝光物品 / 总物品数
    Coverage float64  // > 0.6 为好

    // 新颖性
    // 推荐物品中用户未见过物品比例
    Novelty float64  // > 0.3 为好
}
```

#### 2. DPP 多样性重排

```go
func DPPReRank(items []RankItem, lambda float64) []RankItem {
    // DPP：在保证相关性的前提下，最大化多样性
    // 边际效用 = relevance_score + lambda × diversity_gain
    // 贪心选择使边际效用最大的物品
}
```

#### 3. 曝光去重

```
规则：
  - 用户最近 N 天看过的内容不再推荐
  - 同一作者内容最多连续出现 2 次
  - 同一标签内容不能超过 30%
```

#### 4. Exploration 机制

```
Epsilon-Greedy：
  - epsilon = 0.1（10% 概率随机推荐）
  - 保证一定 Exploration

UCB（Upper Confidence Bound）：
  - 不确定性高的物品优先推荐
  - 自然平衡 Exploration 和 Exploitation
```

## F3: 推荐延迟过高

### 场景

| 问题 | 表现 | 根因 |
|------|------|------|
| **特征获取慢** | P99 > 50ms | Redis 未命中，请求串行 |
| **模型推理慢** | P99 > 30ms | 模型过大，CPU 限制 |
| **向量检索慢** | P99 > 20ms | FAISS 索引未优化 |
| **召回量大** | 候选集 10W，延迟高 | 计算量太大 |

### 应对策略

#### 1. 特征获取优化

```
并行拉取：
  - 用户基础特征 → 并行 → 用户实时兴趣 → 并行 → 物品特征
  - 而不是串行

缓存热点：
  - 热点用户（高活跃）缓存 30s
  - 热点物品特征缓存 5min

降级策略：
  - 特征获取超时 → 用默认值
  - 不阻塞推荐流程
```

#### 2. 模型推理优化

```
方案 1：模型蒸馏
  - 大模型（DeepFM 100M 参数）→ 小模型（DeepFM 10M 参数）
  - 精度损失 < 5%，推理速度提升 5x

方案 2：模型量化
  - float32 → int8
  - 内存降低 4x，推理速度提升 2-3x

方案 3：模型剪枝
  - 删除不重要的神经元
  - 参数减少 30-50%，速度提升

方案 4：GPU 加速
  - 使用 TensorRT / CUDA 加速
  - P99 从 30ms 降到 5ms
```

#### 3. 向量检索优化

```
FAISS 优化：
  - IndexHNSW（nprobe=16）→ 高速但内存大
  - IndexIVFFlat + Flat（nprobe=50）→ 准且快
  - 多核并行计算

分层索引：
  - 热内容：IndexFlat（精确检索）
  - 冷内容：IndexHNSW（近似检索）
```

## F4: 数据漂移（Data Drift）

### 场景

| 类型 | 表现 | 根因 |
|------|------|------|
| **概念漂移** | CTR 明显变化，模型不再适配 | 用户兴趣变化 |
| **协变量漂移** | 特征分布变化 | 流量构成变化、季节性 |
| **先验漂移** | 物品分布变化 | 内容更新、热点变化 |

### 应对策略

#### 1. 模型定期更新

```
更新频率：
  - 全量训练：每天
  - 增量训练：每小时
  - 实时更新：分钟级（针对强信号）

监控指标：
  - 模型 AUC < 0.65 → 触发重新训练
  - 特征分布 PSI > 0.2 → 触发特征重采样
```

#### 2. 在线学习

```go
type OnlineLearner struct {
    model *FTRL  // Follow The Regularized Leader
}

func (o *OnlineLearner) Update(sample *Sample) {
    // 每次收到反馈立即更新
    // 不需要批量训练
    o.model.IncrementalUpdate(sample)
}
```

#### 3. 概念漂移检测

```go
func DetectDrift(metrics []WindowMetric) bool {
    // CUSUM 检测
    // 计算累计和，当超过阈值则判定漂移

    recent := computeMean(metrics[-100:])
    historical := computeMean(metrics[:-100])

    if abs(recent - historical) / historical > 0.1 {
        return true  // 漂移检测到
    }
    return false
}
```

## F5: 特征失效

### 场景

| 问题 | 表现 | 根因 |
|------|------|------|
| **用户特征过时** | 推荐结果不反映最新兴趣 | 短期兴趣未更新 |
| **物品特征过时** | 热度分不反映真实情况 | 统计量更新延迟 |
| **上下文特征缺失** | 推不出符合场景的内容 | 时间/位置特征错误 |

### 应对策略

#### 1. 特征 TTL 管理

```
短期兴趣（5-30 分钟过期）：
  - 用户点击/播放行为
  - Redis key TTL = 30min

中期特征（1-24 小时过期）：
  - 用户兴趣标签
  - Redis key TTL = 6h（每小时刷新）

长期特征（天级更新）：
  - 用户基础画像
  - MySQL，天级批更新
```

#### 2. 特征新鲜度监控

```prometheus
# 特征延迟
feature_freshness_seconds{type="user_interest"} 180  # 3分钟前更新
feature_freshness_seconds{type="item_stats"} 600     # 10分钟前更新

# 告警：特征延迟 > 30min
```

#### 3. 降级策略

```
特征获取超时时：
  - 使用历史快照（允许短暂不一致）
  - 不能使用默认值（如 age=0）
  - 记录告警，等待特征恢复
```

## F6: 模型效果下降

### 场景

| 问题 | 表现 | 根因 |
|------|------|------|
| **模型退化** | AUC 持续下降 | 训练数据和测试数据分布不一致 |
| **反馈循环** | 推荐越来越集中 | 协同过滤的强化效应 |
| **偏向热门** | 模型倾向于推荐热门内容 | 热门内容正样本多 |

### 应对策略

#### 1. 模型效果监控

```prometheus
# 天级 AUC（验证集）
model_auc{version="v3.2.1"} 0.7823
model_auc{version="v3.2.0"} 0.7856
# 如果连续 3 天 AUC 下降 > 1%，触发告警

# 在线 CTR 预估
ctr_pred_avg{version="v3.2.1"} 0.0432
ctr_pred_avg{version="v3.2.0"} 0.0428
```

#### 2. 反馈循环检测

```go
func DetectFeedbackLoop() {
    // 检测推荐集中度
    coverage := computeCoverage()  // 每天曝光物品数 / 总物品数

    if coverage < 0.4 {
        // 推荐集中度过高，可能是反馈循环
        alert("Coverage too low, possible feedback loop")
    }

    // 检测协同过滤强化效应
    cfEnhancement := computeCFEnhancement()  // CF 推荐内容占比

    if cfEnhancement > 0.7 {
        alert("CF enhancement too high, might cause filter bubble")
    }
}
```

#### 3. 训练数据去偏

```
曝光偏差：
  - 使用 IPS（Inverse Propensity Score）加权
  - 曝光多但未点击的样本降低权重

热门偏差：
  - 对热门物品降采样
  - 平衡热门和长尾样本比例

位置偏差：
  - 使用 Position Debiasing 模型
  - 位置特征不参与训练，只用于预测
```

## F7: AB 实验结果不置信

### 场景

| 问题 | 表现 | 根因 |
|------|------|------|
| **流量不均** | 实验组/对照组流量比例 45%/55% | 分桶不均匀 |
| **新奇效应** | 实验组初期 CTR 虚高 | 用户好奇新功能 |
| **Seasonal Effect** | 工作日/周末效果差异大 | AB 实验时间选择不当 |
| **样本量不足** | 置信区间过宽 | 流量太小 |

### 应对策略

#### 1. 流量分层隔离

```
层 1：推荐模型实验
层 2：重排策略实验
层 3：UI 展示实验

每层流量正交：
  - 层 1 分桶：hash(user_id + layer_id) % 100
  - 层 2 分桶：hash(user_id + layer_id) % 100
  - 用户同时只在一个层中
```

#### 2. 新奇效应处理

```
观察期（Burn-in Period）：
  - 实验开始前 3-5 天数据不计入
  - 等新奇效应消退

双盲验证：
  - 新功能不告知用户
  - 减少新奇效应影响
```

#### 3. 实验结果检验

```go
type ABResult struct {
    Variant         string
    SampleCount     int64
    MeanValue       float64
    Variance        float64
    CI_Lower        float64
    CI_Upper        float64
    PValue          float64
    Significant     bool
}

func AnalyzeABResult(control, treatment []float64) ABResult {
    // t-test 计算显著性
    // 95% 置信区间

        PValue: pValue,

}
```

## F8: 推荐系统故障导致服务不可用

### 场景

| 问题 | 影响 | 恢复时间 |
|------|------|----------|
| **推荐服务崩溃** | 无法返回推荐 | < 1min（重启）|
| **特征服务不可用** | 无法获取用户/物品特征 | < 5min |
| **模型服务不可用** | 无法排序 | < 1min（降级到规则）|
| **Kafka 堆积** | 实时特征停止更新 | < 30min |

### 应对策略

#### 1. 熔断降级

```
推荐服务不可用时：
  1. 返回热门内容兜底（Home Feed Default）
  2. 用户无感知（延迟正常）
  3. 记录告警，事后排查

特征服务不可用时：
  1. 使用最后一次缓存的特征
  2. 允许短暂不一致
  3. 特征恢复后自动更新
```

#### 2. 兜底策略

```
推荐降级路径：
  1. 正常：多路召回 → 精排 → 重排 → 返回
  2. 精排降级：多路召回 → 规则排序 → 返回
  3. 召回降级：热门召回 → 规则排序 → 返回
  4. 全量降级：热门内容 → 直接返回

兜底内容：
  - 全局热门内容
  - 用户类别偏好热门
  - 最后一次成功推荐缓存
```

#### 3. 高可用架构

```
推荐服务多节点部署：
  - 每个服务 3+ 节点
  - 负载均衡
  - 自动故障转移

特征服务多节点：
  - Redis Cluster
  - MySQL 主从
  - 本地缓存兜底

监控和告警：
  - 延迟 P99 > 200ms → P2
  - 成功率 < 99% → P1
  - 服务崩溃 → P0
```

## F9: 恶意行为干扰推荐

### 场景

| 类型 | 表现 | 影响 |
|------|------|------|
| **刷量** | 虚假点击/播放 | 模型学到错误反馈 |
| **SEO 优化** | 标题党/封面党 | CTR 虚高，质量下降 |
| **套现** | 诱导点击广告 | 商业指标受损 |
| **钓鱼** | 欺诈内容推荐 | 平台安全 |

### 应对策略

#### 1. 反作弊系统

```
实时检测：
  - IP 聚集度（同一 IP 大量行为）
  - 设备聚集度（同一设备大量账号）
  - 行为模式（固定间隔点击）
  - 设备farm

处理：
  - 实时过滤（不计入推荐）
  - 拉黑（账号封禁）
  - 降权（减少推荐量）
```

#### 2. 内容质量控制

```
前置过滤：
  - 低质量物品直接过滤（quality_score < 0.2）
  - 成人内容不过审
  - 标题党检测（NLP 模型）

后置检验：
  - 用户举报内容降权
  - 完播率低的物品降低推荐
  - 负面评论多的物品降低推荐
```

#### 3. 反馈信号清洗

```
清洗规则：
  - 异常 IP 的行为不计入训练样本
  - 异常设备的用户行为降低权重
  - 诱导内容的点击不计入正样本

监控指标：
  - 刷量检测率 > 95%
  - 清洗后样本质量
```
