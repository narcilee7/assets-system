# Interview

## 核心追问与权衡讨论

### 1. WebSocket 和 SSE 如何选择？

#### 面试官想知道

- 你是否理解两个协议的本质区别
- 你是否能根据业务场景选择合适的协议
- 你是否理解各自的优缺点和适用场景

#### 讨论要点

```
核心区别：

WebSocket：
  - 全双工，TCP 连接升级后保持打开
  - 可发送二进制和文本
  - 需要自行实现心跳、重连
  - 企业网络可能不支持（需要 HTTP Upgrade）

SSE（Server-Sent Events）：
  - 半双工，基于 HTTP/1.1
  - 只能是文本（text/event-stream）
  - 浏览器原生支持自动重连
  - 容易被企业防火墙穿透

选择原则：
  ┌────────────────────────────────────────────────────┐
  │ 场景：聊天、游戏、协作 → WebSocket（全双工，低延迟）│
  │ 场景：Agent 流式输出、实时看板 → SSE（单向，简单）  │
  │ 场景：企业内网 → SSE（HTTP 友好）                   │
  │ 场景：需要传输二进制（文件、图片）→ WebSocket      │
  └────────────────────────────────────────────────────┘
```

#### 追问

**Q：什么时候用 Long Polling？**

A：
- SSE 和 WebSocket 都不支持时（极端企业网络限制）
- 实现最简单，但性能最差（每次请求都要 HTTP 握手）
- 只能作为最后的降级方案

### 2. 如何保证消息可靠送达？

#### 面试官想知道

- 你是否理解"可靠送达"的层次（TCP 可靠 ≠ 应用层可靠）
- 你是否有完整的 ACK 机制设计
- 你是否理解离线消息的处理方式

#### 讨论要点

```
可靠送达的层次：

1. TCP 层：TCP 自身保证可靠传输
   - 但只保证数据链路层，客户端收到 ≠ 应用层收到

2. 应用层 ACK：
   - 服务端发消息 → 客户端回 ACK → 服务端才认为送达
   - 如果 30s 内没收到 ACK → 保留为离线消息

3. 消息持久化：
   - 消息先落 MySQL，再投递
   - 服务端重启，已持久化的消息不会丢失

4. 幂等去重：
   - 客户端按 mid 去重
   - 重连补发的消息，不会出现重复显示

5. 离线补发：
   - 用户上线时，从 MySQL 查出 pending 消息补发
```

#### 追问

**Q：如果客户端收到消息后 ACK 没送达服务端（网络抖动），怎么办？**

A：
```
场景：服务端发了 M1，客户端收到但 ACK 在网络抖动中丢失

时序：
  1. 服务端发 M1（pending）
  2. 客户端收到 M1，显示给用户
  3. 客户端回 ACK（M1）
  4. ACK 丢失，服务端 30s 内没收到 ACK
  5. 服务端认为 M1 未送达，用户下次上线会补发 M1

问题：用户会看到重复的 M1

解决：
  - 客户端收到消息后立即本地标记（本地已读）
  - 即使补发 M1，本地去重后不显示
  - 需要本地持久化 last_received_mid
```

### 3. 如何处理消息乱序？

#### 面试官想知道

- 你是否理解乱序的原因（网络抖动、重连补发、多设备）
- 你是否有基于 seq 的排序方案

#### 讨论要点

```
乱序场景：

1. 网络抖动：M1 延迟，M2 先到
2. 重连补发：M1 在补发窗口内，M2 是实时消息
3. 多设备：设备 A 和 B 收到消息顺序不一致

解决方案：客户端按 seq 排序

实现：
  - 服务端每条消息带 seq（房间内严格递增）
  - 客户端本地维护排序队列
  - 收到消息按 seq 插入正确位置
  - 显示时从队列头部依次取出

注意事项：
  - seq 只保证同一房间内有序
  - 跨房间消息可能乱序（但通常没关系）
```

#### 追问

**Q：如果 seq 出现断层（1, 2, 4），怎么办？**

A：
```
场景：收到 seq=1, 2, 4（缺少 seq=3）

可能原因：
  - M3 还在网络中，稍后会收到
  - M3 丢失（网络丢包）

处理策略：
  - 等待 2s，看是否能收到 M3
  - 2s 后还没收到 M3：按当前顺序显示，标记可能丢失
  - 后续收到 M3：插入到正确位置

告警：
  - 如果频繁出现 seq 断层，需要检查网络质量
```

### 4. 长连接如何检测断连？

#### 面试官想知道

- 你是否理解心跳机制的设计原理
- 你是否理解中间设备（NAT、代理）对长连接的影响

#### 讨论要点

```
断连检测机制：

1. TCP keepalive（操作系统层）
   - 优点：不用应用层参与，检测网络分区
   - 缺点：检测时间长（默认 2 小时），不灵活

2. 应用层心跳（推荐）
   - 服务端每 15s 发 Ping
   - 客户端 5s 内回 Pong
   - 连续 3 次无响应 → 判定断连

心跳策略：
  - 间隔必须 < 中间设备超时时间（NAT 通常 60s+）
  - 需要随机 jitter（避免所有连接同时超时）
  - WebSocket 用 Ping/Pong 帧，SSE 用 event: heartbeat
```

#### 追问

**Q：心跳间隔太短有什么问题？**

A：
- 浪费带宽和服务器资源（每秒成千上万次心跳）
- 可能被某些安全设备误判为攻击
- 但总体来说利大于弊，建议 15-30s 为宜

**Q：客户端如何检测断连？**

A：
- 客户端心跳计时器，45s 没收到消息（或没收到 Pong）→ 判定断连
- WebSocket 的 `onclose` 事件（连接真正关闭时触发）
- `onerror` 事件（错误发生）

### 5. 10W 人同时在的房间，如何高效广播消息？

#### 面试官想知道

- 你是否理解广播的性能瓶颈
- 你是否有分层广播的设计思路

#### 讨论要点

```
原始方案（O(N)）：
  for conn in room.connections:
      conn.Send(msg)

问题：
  - 100W 连接 = 100W 次 Write 系统调用
  - CPU 和网络都被打满

优化方案 1：分批写入
  - 每批 1000 个连接，用一次 writev 系统调用
  - 100W 连接 = 1000 批 = 1000 次系统调用

优化方案 2：节点内广播 + 跨节点 Pub/Sub
  - 本地连接：直接发送（本地节点）
  - 远程连接：通过 Redis Pub/Sub 转发

优化方案 3：大房间分片
  - 100W 成员的房间拆成 100 个 1W 成员的小房间
  - 减少单次广播的连接数

实际生产中通常组合使用：分批 + 跨节点 Pub/Sub
```

#### 追问

**Q：如果有 1W 个这样的大房间同时广播呢？**

A：
- 这是典型的热点问题
- 解决方案：消息队列削峰
  - 广播不是同步发送，而是扔到消息队列
  - 后台 worker 慢慢处理
  - 用户不会立刻收到，但最终都能收到
- Rate limit：限制每节点每秒广播的消息数

### 6. 如何设计流式输出（Agent 场景）？

#### 面试官想知道

- 你是否理解流式输出的完整流程
- 你是否能处理断连后流状态恢复

#### 讨论要点

```
流式输出流程：

1. 用户发起请求，Agent 开始生成
2. 服务端创建流任务（stream_id）
3. LLM 每输出一个 token，服务端立即推送 stream_data
4. 用户看到文字逐字出现（流式效果）
5. LLM 输出完毕，发送 stream_end

断连恢复：
  - 流开始时，每个 token 都写入 Redis 缓冲
  - 用户断连重连后，从 Redis 获取已缓冲的内容
  - 继续从断连处接收新 token

关键点：
  - 不能等 LLM 输出完再发，要立即推送每个 token
  - Redis 缓冲防止断连丢失已生成的内容
  - stream_id 要全局唯一（Snowflake）
```

#### 追问

**Q：如果 Redis 不可用，流式输出的缓冲怎么办？**

A：
- 降级方案：内存缓冲 + 定期刷盘
  - 每个 token 先存内存
  - 每 10 个 token 异步刷到 MySQL
  - 断连恢复时，从 MySQL 读取已缓冲的内容
- 注意：降级会轻微增加延迟，但保证可靠性

### 7. 如何设计断线重连的补发机制？

#### 面试官想知道

- 你是否理解 Last-Event-ID 的工作原理
- 你是否能处理补发过程中的各种边界情况

#### 讨论要点

```
SSE 的补发机制（原生支持）：

客户端：
  - 断开重连时，带上 X-Last-Event-ID: {最后收到的消息ID}
  - 服务端自动补发 id > Last-Event-ID 的所有消息

WebSocket 的补发（需要自己实现）：

1. 连接建立时，服务端主动查询离线消息
   - SELECT * FROM messages WHERE mid > {last_event_id} AND to_user = {user_id}
   - 按顺序补发

2. 客户端按 mid 去重
   - 如果 mid 已存在，丢弃（防止重复显示）

3. 补发完成后，发送 reconnect 事件
   - 告知客户端补发了多少条消息
```

#### 追问

**Q：如果 Last-Event-ID 之后有 100W 条消息，怎么办？**

A：
```
场景：用户离线 1 小时，有 100W 条消息

处理策略：
  1. 分页补发：每次补发 1000 条，补完为止
  2. 或者只补发最近 1000 条（按业务需求）
  3. 或者用时间窗口：从上次上线时间开始补发

业务权衡：
  - 补发太多，用户等待时间长，体验差
  - 补发太少，用户可能错过重要消息
  - 通常策略：只补发最近 24h 的消息
```

### 8. 连接数如何水平扩展？

#### 面试官想知道

- 你是否理解长连接场景下的扩展挑战
- 你是否有完整的架构设计

#### 讨论要点

```
扩展方案：

1. Load Balancer 层
  - L4 LB（TCP 层）按 user_id hash 路由
  - 保证同一用户的请求打到同一节点（会话保持）
  - 也可以用一致性 hash（节点变更时影响小）

2. 节点内连接表
  - 每节点维护本地连接表（sync.Map）
  - 热点用户连接信息缓存本地
  - 冷数据查 Redis

3. 跨节点消息路由
  - 消息路由到用户所在节点
  - 通过 Redis Pub/Sub 跨节点通信

4. 状态外置
  - 连接状态存 Redis，不存节点内存
  - 节点故障，用户可连接到其他节点
  - 缺点：每次都要查 Redis

推荐方案：本地缓存 + Redis 外置 + 消息队列削峰
```

#### 追问

**Q：如何保证同一用户的多个设备请求打到同一节点？**

A：
- 如果使用 L4 LB，按 user_id hash 路由，同一用户会打到同一节点
- 但多设备意味着同一人有多个连接，可能在不同节点
- 解决方案：消息按 user_id 路由，而不是按 conn_id
  - 无论用户有几个设备，只要有一个在线，就投递给在线的连接
  - 离线设备的消息暂存离线表，等用户下次上线补发

### 9. 生产环境踩过的坑

#### 面试官想知道

- 你是否有真实的故障处理经验
- 你是否能从故障中总结系统性改进

#### 典型案例

```
案例 1：10W 连接同时断开的惊群效应

现象：服务器升级重启，10W 用户同时断连，然后同时重连
       服务器在 10s 内收到 10W 次连接请求，直接打挂

根因：
  - 心跳超时时间固定，10W 连接几乎同时超时
  - 重连没有退避，全部立即重连

解决：
  1. 心跳加随机 jitter（不同连接在不同时间超时）
  2. 服务端升级前，先通知客户端预计断连时间
  3. 客户端实现指数退避重连
  4. 服务端配置连接速率限制

教训：长连接场景下，所有"同时发生"的事件都要有错峰机制

---

案例 2：热点房间消息延迟 10s

现象：明星直播房间 100W 成员，消息延迟 10s 才送达

根因：
  - 每次消息广播要遍历 100W 连接
  - 单节点 CPU 打满，消息队列积压

解决：
  1. 大房间拆分成多个子房间（room-live-001 ~ room-live-100）
  2. 成员均匀分散到 100 个子房间
  3. 每子房间 1W 成员，单次广播只需处理 1W 连接
  4. 消息通过广播层聚合，复制到各子房间

教训：大房间是实时系统的主要挑战，需要提前识别并做架构调整

---

案例 3：Redis 单节点内存打满

现象：Redis 内存达到 64GB 上限，连接状态写入失败

根因：
  - 连接 key 没有设置 TTL，或者 TTL 太长
  - 断连后连接状态没及时清理
  - 历史数据（离线消息）堆积

解决：
  1. 连接 key 必须设置 EXPIRE（60s 心跳续命）
  2. 断连时立即删除 Redis 连接 key
  3. 离线消息按时间清理（保留 7 天）
  4. 迁移到 Redis Cluster 分片

教训：Redis 内存要时刻监控，key 生命周期要严格管理
```
