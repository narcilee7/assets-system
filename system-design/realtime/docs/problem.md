# Problem

## 需求

构建一个生产级的实时通信系统，支持服务器向客户端推送消息、处理海量并发长连接、保障消息可靠送达。

### 功能需求

#### 1. 消息推送（核心能力）

- **服务端推送**：服务器主动向已连接客户端发送消息（聊天、通知、实时数据更新）
- **客户端推送**：客户端向服务器发送消息（上行数据）
- **双向通信**：全双工通道（WebSocket），或半双工 + 请求驱动（SSE）
- **消息广播**：一条消息可同时推送给多个订阅者（主题/房间）

#### 2. 连接管理

- **长连接维护**：客户端与服务器保持持久 TCP 连接
- **心跳机制**：主动探测连接存活状态，防止连接被中间设备误关
- **断线重连**：网络波动后自动重建连接，恢复会话
- **连接迁移**：用户从设备 A 切换到设备 B，消息如何同步

#### 3. 消息可靠性

- **消息持久化**：离线用户的消息需要存储，用户上线后补发
- **消息确认（ACK）**：客户端收到消息后回 ACK，服务端才知道已送达
- **消息去重**：重连时可能收到重复消息，需要业务层幂等处理
- **消息时序**：多端同步时，后发送的消息不能先显示（乱序处理）

#### 4. 协议选择

| 协议 | 双向性 | 穿越防火墙 | 浏览器原生支持 | 实现复杂度 | 适用场景 |
|------|--------|-----------|--------------|-----------|---------|
| **WebSocket** | 全双工 | 需 HTTP Upgrade | 是 | 中 | 聊天、协作、游戏、实时数据 |
| **SSE** | 半双工 | 容易（HTTP） | 是（除 Safari） | 低 | 推送通知、实时数据看板、Agent 流式输出 |
| **Server-Sent Events** | 单通道 | 容易 | 是 | 低 | 单向推送，WebSocket 不必要时 |
| **gRPC Streaming** | 全双工/单向 | 需 HTTP/2 | 需 gRPC-Web | 高 | 微服务间流，后端到后端 |

#### 5. 房间 / 主题模型

- **房间（Room）**：用户加入房间，接收该房间内所有人的消息（如聊天群）
- **主题（Topic）**：基于主题订阅，发布者发送到主题，所有订阅者收到（如系统通知）
- **用户直接消息（DM）**：点对点消息，通过用户 ID 路由

#### 6. 流式输出（Streaming）

- **Agent 流式输出**：大语言模型一个字一个字地输出，需要实时流式传输（SSE）
- **下载进度**：大文件下载的实时进度推送
- **数据管道**：上行数据流 + 下行数据流同时存在

### 非功能需求

| 指标 | 目标 | 说明 |
|------|------|------|
| 同时在线连接数 | 100W+ | 单节点，长连接内存占用是关键 |
| 消息延迟（P99） | < 100ms | 同一区域，端到端延迟 |
| 消息送达率 | 99.9% | 含重连补发 |
| 断连恢复时间 | < 3s | 断线重连到收到消息 |
| 心跳间隔 | 15-30s | 太短浪费资源，太长发现慢 |
| 消息顺序保证 | 基本有序 | 允许因网络重传导致的小乱序 |
| 系统可用性 | 99.95% | 连接管理服务必须高可用 |

---

### 约束

- **消息不能跨区域**：用户必须连接到就近的边缘节点，跨区域消息通过内部网关转发
- **长连接有状态**：连接信息必须在节点内存或分布式缓存中，无法完全无状态
- **消息送达保证 ≠ 消息已读**：用户看到消息才算真正送达，需要客户端 ACK
- **移动网络特殊性**：4G/5G 下连接可能被 NAT 超时断开，需要更频繁的心跳
- **WebSocket 受限场景**：某些企业网络不允许 WebSocket，此时降级为 SSE 或 Long Polling

### 关键设计决策

#### 决策 1：WebSocket vs SSE vs Long Polling

| 维度 | WebSocket | SSE | Long Polling |
|------|-----------|-----|-------------|
| 协议基础 | TCP Socket | HTTP/1.1 | HTTP/1.1 |
| 双向通信 | ✅ 全双工 | ❌ 仅服务器推送 | ⚠️ 伪全双工（每次请求即一次往返）|
| 连接数开销 | 1 个 TCP + 1 个 WS | 1 个 HTTP | 1 个 HTTP（频繁新建）|
| HTTP/2 兼容 | ✅（HTTP/2 上复用）| ✅ | ❌（每次都是新请求）|
| 自动重连 | ❌ 需自行实现 | ✅ 原生支持 | ✅ 原生支持 |
| 数据格式 | 二进制/文本 | 文本（text/event-stream）| 文本或二进制 |
| 浏览器支持 | 全部 | 全部（除 Safari 早期版本）| 全部 |
| 实现复杂度 | 中 | 低 | 中 |
| 穿越企业防火墙 | 困难（需白名单）| 容易 | 容易 |

**场景选择**：
- **聊天 / 游戏 / 协作工具**：WebSocket（全双工，低延迟）
- **Agent 流式输出 / 实时看板 / 通知推送**：SSE（单向，简单，HTTP 友好）
- **企业内网 / 不支持 WebSocket 的环境**：SSE 降级或 Long Polling

#### 决策 2：连接状态存储在哪里？

| 方案 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| **进程内存** | 每个节点维护自己的连接表 | 零网络开销，最快 | 节点故障连接丢失，无法水平扩展 |
| **Redis Pub/Sub** | 连接信息存 Redis，所有节点订阅 | 跨节点广播简单 | Redis 成为瓶颈（连接数限制）|
| **Redis Hash** | 每连接一个 key，TTL 管理 | 状态外置，可水平扩展 | 每次消息都要查 Redis |
| **分布式缓存 + 本地缓存** | 本地热数据 + Redis 兜底 | 本地读取快，Redis 做一致性保障 | 实现复杂度高 |
| **CRDT 状态同步** | 节点间同步连接状态 | 无中心节点 | 实现复杂，适用于特定场景 |

**推荐**：本地缓存 + Redis 外置（热点连接在本地，跨节点消息通过 Redis Pub/Sub）

#### 决策 3：消息如何路由到用户？

```
场景：服务器要给用户 U 发送消息 M

方案 A：按连接路由
  - 消息入口查询：用户 U 当前连接到哪个节点？
  - 直接投递给该节点
  - 优点：延迟最低
  - 缺点：连接感知必须是分布式的

方案 B：统一消息队列
  - 所有消息发到 MQ（Kafka / RabbitMQ）
  - 消息带上目标用户 ID
  - 各节点消费自己负责的用户的消息
  - 优点：解耦，水平扩展简单
  - 缺点：多一跳延迟（< 5ms）

方案 C：混合（推荐）
  - 同一房间的消息 → 广播到所有节点（Pub/Sub）
  - 点对点消息 → 按连接路由（Redis 查用户节点）
  - 离线消息 → 持久化到存储，用户上线后补发
```

#### 决策 4：消息 ID 如何设计？

```
设计目标：
  1. 全局唯一（跨节点、跨时间不冲突）
  2. 趋势递增（用于排序和去重）
  3. 包含时间信息（可从 ID 反推时间）

方案：Snowflake + 用户 ID
  ID = timestamp[41bits] + node_id[10bits] + sequence[12bits]
  - timestamp: 毫秒级时间戳（可表示 69 年）
  - node_id: 节点编号（解决时钟回拨）
  - sequence: 同一毫秒内的递增序号

特点：
  - 单节点内严格递增
  - 跨节点通过 node_id 保证不冲突
  - 可从 ID 反推消息时间（用于时序展示）
  - 缺陷：时钟回拨时 sequence 复用 → 需要在 sequence 中混入随机数
```

### 真实案例与踩坑

#### 案例 1：微信 NTN 切换导致的消息延迟

微信用户从 4G 切换到 WiFi 时，TCP 连接在旧网络上已经 FIN，但客户端还未感知。客户端继续往旧连接发消息，导致消息延迟 30s+ 才送达。

**教训**：
- 心跳间隔必须短于 NAT 超时时间（通常 30-60s）
- 切换网络后要立即重新建立连接，而不是等待心跳探测失败
- 消息要有时间戳，客户端按时间戳排序而非按到达顺序

#### 案例 2：聊天消息顺序错乱

用户 A 在地铁里信号不好，消息 M1 发送后 30s 才送达服务器。同时用户 A 发了 M2（已送达），导致 M2 显示在 M1 前面（乱序）。

**教训**：
- 消息 ID 携带逻辑时间戳（Lamport Clock 或单调递增序号）
- 客户端收到消息后，按 ID 排序展示，而非按到达顺序
- 服务端需要检测乱序，自动补偿（按 ID 顺序重排后再推送）

#### 案例 3：SSE 连接数爆表

某系统使用 SSE 长连接推送实时数据，10W 用户在线，但服务器只开了 1W 连接限制（默认 Linux 单进程文件描述符限制）。所有连接都建在少数几个进程上，导致连接数迅速打满。

**教训**：
- Linux 默认 `ulimit -n` 约为 1024，企业配置通常 4096-65535
- 长连接场景必须调高 `ulimit -n`（建议 > 100W 时设为 100W）
- 也可以用 SO_REUSEPORT 多进程均衡连接

#### 案例 4：AWS ALB 对 WebSocket 的 4 分钟超时

AWS ALB 默认对 WebSocket 连接 4 分钟无流量就断开（即使底层 TCP 是长连接）。客户端和服务端都不知道连接已被 ALB 切断，继续在该连接上发消息，导致消息丢失。

**教训**：
- WebSocket 也需要心跳（即使 TCP 连接活着，中间代理可能超时）
- 心跳间隔 < 4 分钟（建议 15-30s）
- 或者配置 ALB 超时时间为更长（但不是所有场景都能改）
