# Problem

## 需求

构建一个生产级订单系统，处理下单、库存扣减、支付、物流等完整流程，保证数据一致性和系统可用性。

### 功能需求

#### 1. 订单状态机

```
待支付 → 已支付 → 已发货 → 已完成
   ↓
已取消（超时/用户取消）
   ↓
已退款
```

| 状态 | 说明 |
|------|------|
| 待支付 | 库存已预占，等待用户支付 |
| 已支付 | 支付成功，准备发货 |
| 已发货 | 物流已发出 |
| 已完成 | 用户确认收货 |
| 已取消 | 超时/主动取消，库存回滚 |
| 已退款 | 退款完成 |

#### 2. 库存管理

| 策略 | 实现 | 适用场景 |
|------|------|-----------|
| **预占库存** | 下单时预占库存，支付后真正扣减 | 秒杀、抢购 |
| **下单即扣减** | 下单直接扣减库存 | 普通电商 |
| **支付后扣减** | 支付成功才扣减 | 高价值商品 |

#### 3. 超时取消

| 场景 | 实现 |
|------|------|
| 订单超时 | 定时任务扫描待支付订单 |
| 库存超时释放 | 订单取消时回滚库存 |
| 支付超时 | 支付渠道回调超时 |

---

### 关键设计决策

#### 决策 1：库存预占还是直接扣减？

| 策略 | 优点 | 缺点 |
|------|------|------|
| **预占库存** | 保留库存给用户，无超卖 | 实现复杂 |
| **直接扣减** | 简单 | 可能超卖 |

**推荐**：**预占库存**，防超卖是电商核心。

#### 决策 2：库存回滚时机？

| 时机 | 实现 | 风险 |
|------|------|------|
| 订单取消时回滚 | 简单 | 库存占用时间不确定 |
| 定时扫描回滚 | 资源利用率高 | 延迟 |
| 支付成功直接扣减 | 零回滚 | 需要分布式事务 |

**推荐**：**支付成功才真正扣减**，预占库存仅标记。

---

### 真实踩坑

#### 案例 1：库存超卖

秒杀 100 件，1000 人下单成功。

根因：并发下单未正确扣减。

教训：库存扣减必须用乐观锁/分布式锁。

#### 案例 2：订单状态机错乱

用户支付后系统崩溃，订单状态未知。

教训：状态机必须有持久化，状态变更有日志。

#### 案例 3：重复下单

用户双击下单按钮，重复创建两个订单。

教训：幂等下单，token 或唯一键约束。
