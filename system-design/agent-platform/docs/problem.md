# Problem

## 需求

构建一个生产级的 Agent 平台，让 AI Agent 能够理解用户意图、规划任务、调用工具、与外部世界交互，并输出可靠的结果。

### 功能需求

#### 1. 会话管理（Session Management）

会话管理是 Agent 平台的基础，负责维护用户与 Agent 之间的对话上下文。

| 功能 | 描述 |
|------|------|
| **上下文管理** | 管理多轮对话历史，支持上下文窗口内记忆 |
| **状态持久化** | 支持会话恢复，跨设备同步 |
| **会话分组** | 支持多个并行会话（不同任务） |
| **上下文截断** | 当上下文超过窗口时，智能截断旧消息 |

#### 2. 规划器（Planner）

规划器负责将用户意图分解为可执行的步骤。

| 功能 | 描述 |
|------|------|
| **任务拆解** | 将复杂任务拆解为子任务序列 |
| **约束注入** | 在规划中加入用户约束（时间、成本、质量） |
| **可解释性** | 生成可解释的执行计划，用户可确认 |
| **动态规划** | 根据执行结果动态调整计划 |
| **循环检测** | 检测重复执行，避免死循环 |

#### 3. 工具运行时（Tool Runtime）

工具是 Agent 与外部世界交互的接口。

| 功能 | 描述 |
|------|------|
| **工具定义** | 标准化的工具 schema（名称、描述、参数、返回值）|
| **权限控制** | 基于用户/租户的细粒度工具权限 |
| **执行保障** | 超时控制、重试机制、熔断 |
| **副作用管理** | 识别有副作用工具，避免重复执行 |
| **执行审计** | 记录所有工具调用，用于回溯和合规 |
| **工具编排** | 支持工具间的依赖关系和输出传递 |

#### 4. 记忆系统（Memory）

记忆系统让 Agent 能够跨会话学习用户偏好。

| 类型 | 描述 | 时效 |
|------|------|------|
| **短期记忆** | 当前会话内的上下文（工作记忆）| 会话内 |
| **长期记忆** | 跨会话学习的用户偏好、知识 | 持久化 |
| **情景记忆** | 具体事件的记忆（Episodic Memory）| 持久化 |
| **语义记忆** | 事实性知识（Semantic Memory）| 持久化 |

#### 5. 流式输出（Streaming）

流式输出让 Agent 能够实时返回部分结果。

| 功能 | 描述 |
|------|------|
| **Token 流** | 模型输出逐 token 返回（SSE/WebSocket）|
| **事件流** | 支持多种事件类型（text/code/error/status）|
| **取消支持** | 用户可取消正在执行的 Agent |
| **恢复支持** | 断连后可从断点恢复 |

#### 6. 评估系统（Evaluation）

评估系统用于衡量 Agent 输出质量。

| 功能 | 描述 |
|------|------|
| **Golden Set** | 人工标注的高质量问答对 |
| **Judge LLM** | 用大模型做自动化评估 |
| **回归测试** | 每次变更后的质量回归检测 |
| **指标体系** | 准确率、相关性、安全性等指标 |

#### 7. 安全与权限（Safety）

Agent 执行高风险操作，需要严格的安全控制。

| 功能 | 描述 |
|------|------|
| **数据边界** | 防止 Agent 访问未授权数据 |
| **高风险确认** | 删除、支付等高风险操作需用户确认 |
| **内容过滤** | 输入输出内容安全检测 |
| **租户隔离** | 不同租户的数据和工具完全隔离 |

### 非功能需求

| 指标 | 目标 | 说明 |
|------|------|------|
| 首次响应延迟 | < 500ms | 用户发起请求到收到首 token |
| 流式输出延迟 | < 100ms/token | 模型生成到送达延迟 |
| 工具调用成功率 | > 99.5% | 含重试后的成功率 |
| 工具调用延迟 P99 | < 2s | 不含外部 API 延迟 |
| 会话恢复时间 | < 1s | 断连重连后恢复 |
| 系统可用性 | 99.99% | Agent 服务本身 |
| 评估覆盖率 | > 80% | 核心场景有评估用例 |

### 关键设计决策

#### 决策 1：Agent 执行模式（同步 vs 异步）

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| **同步执行** | 用户等待结果返回 | 简单任务、快速响应需求 |
| **异步执行** | 任务提交后立即返回，后续通过回调/轮询获取结果 | 复杂任务、长时间运行 |
| **流式执行** | 实时返回部分结果（SSE/WebSocket）| 需要实时反馈的场景 |

**推荐**：**流式执行 + 异步回调**（大多数场景）

#### 决策 2：上下文窗口管理

| 方案 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| **固定窗口** | 固定保留最近 N 条消息 | 简单 | 可能丢失重要早期信息 |
| **智能截断** | 根据重要性保留关键消息 | 保留重要信息 | 截断算法复杂 |
| **摘要压缩** | 将历史压缩为摘要 | 保留全量历史 | 压缩有延迟和信息损失 |
| **层级记忆** | 短期→长期→情景→语义逐层抽象 | 信息层次化 | 实现复杂 |

**推荐**：**智能截断 + 层级记忆混合**（根据场景选择）

#### 决策 3：工具调用策略

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| **串行调用** | 按顺序执行工具 | 有依赖关系的工具 |
| **并行调用** | 同时执行无依赖工具 | 无依赖的工具 |
| **条件调用** | 根据前一个结果决定是否调用 | 条件分支 |
| **试探调用** | 先试探性调用，根据结果调整 | 不确定最优工具 |

**推荐**：**并行 + 智能依赖解析**（自动判断并行机会）

#### 决策 4：记忆存储架构

```
短期记忆（Redis）：
  - 会话内上下文
  - TTL = 会话时长 + 缓冲

长期记忆（MySQL/向量数据库）：
  - 用户偏好
  - 知识图谱
  - 向量 Embedding 检索

Memory 检索流程：
  用户查询 → 短期记忆匹配 → 长期记忆语义检索 → 合并上下文
```

### 真实案例与踩坑

#### 案例 1：工具调用死循环

某 Agent 在执行任务时，对一个 API 调用失败后反复重试，最终耗尽配额。

**根因**：没有循环检测和退避策略。

**教训**：
- 工具调用要有最大重试次数限制
- 重试间隔要用指数退避
- 检测到重复模式立即终止

#### 案例 2：上下文溢出导致回答质量下降

用户进行长会话后，Agent 开始给出前后矛盾的答案，因为上下文窗口内的关键信息被截断。

**根因**：固定窗口截断丢失了早期关键信息。

**教训**：
- 智能截断要识别关键信息（实体、约束、偏好）
- 定期将重要信息摘要存入长期记忆
- 监控上下文利用率，提前预警

#### 案例 3：流式输出中断导致任务丢失

用户正在下载文件，连接断开，任务状态丢失，用户不知道是否完成。

**根因**：没有任务持久化和恢复机制。

**教训**：
- 所有任务状态要持久化
- 断连重连后能恢复任务状态
- 支持从断点恢复执行
