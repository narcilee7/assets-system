# Scale

## 性能目标

| 指标 | 目标 | 说明 |
|------|------|------|
| 首次响应延迟 | < 500ms | 用户发起请求到收到首 token |
| 流式输出延迟 | < 100ms/token | 模型生成到送达延迟 |
| 工具调用成功率 | > 99.5% | 含重试后的成功率 |
| 工具调用延迟 P99 | < 2s | 不含外部 API 延迟 |
| 并发会话数 | > 10W | 单节点 |
| 会话恢复时间 | < 1s | 断连重连后恢复 |

## 性能瓶颈分析

### 瓶颈 1：LLM 推理延迟

#### 问题

Agent 输出是流式的，每个 token 生成后需要立即推送，首 token 延迟 + token 间延迟直接影响用户体验。

#### 优化方案

| 方案 | 实现 | 效果 |
|------|------|------|
| **流式输出** | SSE/WebSocket 实时推送 | 立即看到输出 |
| **首 token 优化** | 预测解码、缓存 | 首 token 延迟降低 30% |
| **批量推理** | 多请求合并推理 | 吞吐量提升 |
| **模型蒸馏** | 大模型→小模型 | 推理速度 5x 提升 |

---

### 瓶颈 2：工具调用延迟

#### 问题

工具调用可能涉及外部 API，单次调用延迟 100ms-5s，如果串行执行，总延迟 = sum(工具延迟)。

#### 优化方案

| 方案 | 实现 | 效果 |
|------|------|------|
| **并行工具调用** | 无依赖工具并行执行 | 延迟 = max(工具延迟) 而非 sum |
| **超时熔断** | 单工具超时不影响整体 | 失败隔离 |
| **工具预热** | 提前预热常用工具 | 首次调用加速 |
| **结果缓存** | 相同参数的工具结果缓存 | 重复调用 0 延迟 |

#### 并行工具执行

```go
func (a *Agent) ExecuteToolsParallel(steps []*Step) error {
    // 构建依赖 DAG
    dag := NewDAG(steps)

    // 按层级执行
    for _, level := range dag.Levels() {
        var wg sync.WaitGroup
        for _, step := range level {
            wg.Add(1)
            go func(s *Step) {
                defer wg.Done()
                a.executeTool(s)
            }(step)
        }
        wg.Wait()  // 等待同层所有工具完成
    }

    return nil
}
```

### 瓶颈 3：上下文构建延迟

#### 问题

每次请求需要加载会话历史、检索记忆、构建上下文，延迟 100-500ms。

#### 优化方案

| 方案 | 实现 | 效果 |
|------|------|------|
| **会话状态缓存** | Redis 缓存活跃会话状态 | 加载延迟降低 80% |
| **记忆分层索引** | 短期/长期分层索引 | 检索延迟降低 50% |
| **异步预取** | 下一次请求时预取上下文 | 用户无感知 |
| **上下文压缩** | 将历史压缩为摘要 | 上下文大小降低 70% |

---

### 瓶颈 4：流式输出的服务器推送

#### 问题

SSE/WebSocket 长连接占用服务器资源，高并发时连接数成为瓶颈。

#### 优化方案

| 方案 | 实现 | 效果 |
|------|------|------|
| **连接复用** | HTTP/2 Multiplexing | 连接数降低 |
| **消息批量** | 多个 token 合并发送 | 网络开销降低 |
| **CDN 加速** | 通过 CDN 推送 SSE | 边缘节点分流 |
| **WebSocket 分片** | 按用户分片到不同节点 | 连接分散 |

---

### 瓶颈 5：评估系统计算量

#### 问题

实时评估需要调用 Judge LLM，计算量大，成本高。

#### 优化方案

| 方案 | 实现 | 效果 |
|------|------|------|
| **采样评估** | 只评估有代表性的样本 | 计算量降低 90% |
| **轻量指标** | 用规则代替 LLM | 速度提升 100x |
| **异步评估** | 异步执行，不阻塞主流程 | 无延迟影响 |
| **历史累积** | 评估结果累积，复用 | 减少重复评估 |

## 扩展方案

### 扩展维度 1：多 Agent 并行

```
架构：
  Agent Pool
    ├── Agent-1（处理任务 A）
    ├── Agent-2（处理任务 B）
    └── Agent-N（处理任务 N）

任务分配：
  - 按用户 ID hash
  - 按任务类型分配
  - 按负载均衡

优势：
  - 隔离性好（用户间不共享状态）
  - 容灾好（一个 Agent 崩溃不影响其他）
  - 扩展好（按需增加 Agent 数量）
```

### 扩展维度 2：工具服务化

```
工具执行从 Agent 分离出去：
  ┌─────────────────────────────────┐
  │        Agent 服务               │
  │   - LLM 推理                    │
  │   - 规划                        │
  │   - 流式输出                    │
  └─────────────────────────────────┘
              │ 工具调用请求
              ▼
  ┌─────────────────────────────────┐
  │        工具执行服务              │
  │   - 工具注册与管理               │
  │   - 执行与重试                   │
  │   - 权限控制                    │
  └─────────────────────────────────┘

优势：
  - 工具独立扩展
  - 工具执行环境隔离
  - 工具可以跨 Agent 复用
```

### 扩展维度 3：记忆存储分离

```
短期记忆（Redis）：
  - 会话内上下文
  - 高性能
  - 自动过期

长期记忆（向量数据库）：
  - 跨会话知识
  - 语义检索
  - 持久化

架构分离：
  - 记忆服务独立部署
  - 按需扩展
  - 多 Agent 共享
```

## 容量规划

### 会话容量估算

```
目标并发会话：10W

单个会话资源占用：
  - 内存：~10MB（上下文 + 状态）
  - 连接：1 个 SSE/WebSocket 连接

所需节点数：
  - 单节点 1000 并发会话
  - 所需节点：10W / 1000 = 100 节点
  - 考虑冗余：120 节点

节点配置：
  - CPU：32 核
  - 内存：64GB
  - 网络：10Gbps
```

### LLM 推理容量估算

```
目标 QPS：5W

单节点 LLM 推理吞吐：
  - 批次大小：32
  - 延迟：500ms/批次
  - QPS：32 / 0.5 = 64 QPS

所需节点：
  - 5W / 64 = 782 节点（LLM 推理专用）
  - 或者使用模型服务化（Triton）优化

优化方向：
  - GPU 加速（单卡 1000 QPS）
  - 模型并行
  - 请求合并
```

### 工具执行容量估算

```
工具调用 QPS：2W（假设平均每个会话 0.2 QPS）

单工具执行节点容量：
  - 取决于外部 API 限流
  - 通常 1000 QPS/节点

所需节点：
  - 2W / 1000 = 20 节点
  - 考虑外部 API 限流，需要流量控制

关键：
  - 工具执行异步化
  - 队列削峰
  - 超时保护
```
