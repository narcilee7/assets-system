### 1. Agent 如何规划任务？

#### 面试官想知道

- 你是否理解 Agent 规划的核心机制
- 你是否能处理复杂任务的拆解
- 你是否理解规划的局限性和改进方法

#### 讨论要点

```
任务规划流程：

1. 用户意图理解
   - LLM 分析用户想要什么
   - 识别关键实体和约束

2. 任务拆解
   - 将复杂任务拆解为原子步骤
   - 每个步骤对应一个工具或操作

3. 依赖分析
   - 判断步骤间的依赖关系
   - 确定并行机会

4. 执行计划生成
   - 生成结构化的执行计划
   - 包含步骤、工具、参数、依赖

5. 动态调整
   - 根据执行结果调整后续计划
   - 处理异常情况

规划局限性：
  - LLM 规划可能出错（幻觉）
  - 复杂任务拆解效果差
  - 依赖关系判断可能错误

改进方法：
  - Few-shot prompting 提供示例
  - CoT（Chain of Thought）提升推理
  - 验证步骤可行性再执行
```

### 2. 工具调用失败如何恢复？

#### 面试官想知道

- 你是否理解工具调用的失败模式
- 你是否有完整的错误处理策略
- 你是否能设计可靠的执行机制

#### 讨论要点

```
失败模式：

1. 网络超时
   - 外部 API 网络问题
   - 重试（指数退避）

2. API 限流
   - 请求频率超限
   - 降级或排队

3. 参数错误
   - 工具参数不完整
   - 参数验证

4. 权限问题
   - 用户无权调用
   - 返回明确错误

恢复策略：

1. 重试机制
   - 指数退避（1s → 2s → 4s）
   - 最大重试次数限制

2. 熔断器
   - 连续失败 N 次后熔断
   - 防止资源耗尽

3. 降级策略
   - 工具不可用时降级到替代方案
   - 或返回明确错误让用户知道

4. 死循环防护
   - 检测重复调用
   - 最大调用次数限制
```

### 3. 如何避免重复执行有副作用的工具？

#### 面试官想知道

- 你是否理解工具的副作用概念
- 你是否有幂等性设计
- 你是否能处理并发安全问题

#### 讨论要点

```
副作用工具：
  - 发送邮件
  - 扣款支付
  - 修改数据库
  - 发送通知

幂等性设计：

1. 工具层面
   - 设计幂等操作（如 HTTP DELETE 是幂等的）
   - 使用唯一请求 ID

2. 执行层面
   - 记录已执行的操作
   - 执行前检查是否已执行

3. 副作用防护
   - 标记有副作用的工具
   - 重复执行前需要用户确认

并发安全：
  - 使用分布式锁
  - 乐观锁/悲观锁
  - 幂等表记录操作
```

### 4. 如何设计流式输出的事件协议？

#### 面试官想知道

- 你是否理解流式输出的工程挑战
- 你是否能设计完整的事件协议
- 你是否理解 SSE/WebSocket 的选择

#### 讨论要点

```
事件类型：

1. message_start
   - 消息开始
   - 携带 message_id

2. text_delta
   - 文本增量
   - 携带文本片段

3. tool_call_start
   - 工具调用开始
   - 携带工具名和参数

4. tool_call_end
   - 工具调用结束
   - 携带执行结果

5. error
   - 错误
   - 携带错误码和消息

6. message_end
   - 消息结束
   - 携带统计信息

SSE vs WebSocket：

SSE：
  - 单向（服务端→客户端）
  - 基于 HTTP/1.1
  - 自动重连
  - 简单

WebSocket：
  - 双向
  - 支持更复杂的交互
  - 需要自己处理重连

选择原则：
  - 纯流式输出 → SSE
  - 需要双向交互 → WebSocket
```

### 5. 如何做 Agent 的质量评估？

#### 面试官想知道

- 你是否理解 Agent 评估的复杂性
- 你是否有系统化的评估方法
- 你是否能设计回归测试

#### 讨论要点

```
评估方法：

1. Golden Set 评估
   - 准备高质量问答对
   - 对比 Agent 输出和期望输出
   - 计算准确率

2. Judge LLM 评估
   - 用大模型评估输出质量
   - 多维度打分（准确性、相关性、安全性）
   - 自动化

3. 用户反馈
   - 👍👎 反馈
   - 用户评分
   - 转化率

4. 指标体系
   - 准确率（Accuracy）
   - 相关性（Relevance）
   - 安全性（Safety）
   - 有用性（Helpfulness）

回归测试：
  - 每次变更后运行评估套件
  - 与 baseline 对比
  - 分数下降 > 2% 阻断上线
```

### 6. 如何隔离用户数据和工具权限？

#### 面试官想知道

- 你是否理解多租户安全
- 你是否有完整的数据隔离方案
- 你是否能设计权限模型

#### 讨论要点

```
数据隔离：

1. 会话隔离
   - 每个租户独立的会话存储
   - Redis key 加租户前缀

2. 记忆隔离
   - 用户记忆按 user_id 隔离
   - 检索时只能访问自己的记忆

3. 工具数据隔离
   - 工具只能访问授权的数据
   - 数据边界检查

权限模型：

1. RBAC（基于角色的访问控制）
   - User → Role → Permission
   - 角色：admin / user / developer

2. ABAC（基于属性的访问控制）
   - 基于用户属性、资产属性、环境条件动态决策

3. 工具权限
   - 每个工具定义所需权限
   - 用户必须有该权限才能调用

数据边界检查：
  - 工具执行前检查数据访问权限
  - 防止越权访问
```

### 7. 如何处理 Agent 的上下文长度限制？

#### 面试官想知道

- 你是否理解上下文窗口的限制
- 你是否有智能的上下文管理策略
- 你是否能设计长期记忆系统

#### 讨论要点

```
上下文限制问题：
  - 模型 context 有限（如 8K / 32K / 128K tokens）
  - 会话历史持续增长
  - 超出限制会导致信息丢失

管理策略：

1. 固定窗口截断
   - 保留最近 N 条消息
   - 简单但可能丢失重要早期信息

2. 智能截断
   - 识别关键信息（实体、约束、偏好）
   - 优先保留关键信息
   - 摘要压缩早期信息

3. 分层记忆
   - 短期记忆：当前会话（工作记忆）
   - 长期记忆：跨会话持久化
   - 需要时从长期记忆检索补充

4. 外部化
   - 不把全部历史放 context
   - 存到外部存储，需要时检索
```

### 8. 生产环境踩过的坑

#### 面试官想知道

- 你是否有真实的故障处理经验
- 你是否能从故障中总结系统性改进

#### 典型案例

```
案例 1：工具调用死循环

现象：Agent 在执行任务时反复调用同一个工具，配额在 1 分钟内耗尽

根因：
  - 工具返回结果触发 Agent 重新规划
  - 重新规划后又调用同一工具
  - 没有循环检测机制

解决：
  1. 增加循环检测（连续 3 次调用同一工具 → 终止）
  2. 设置工具调用最大次数
  3. 增加指数退避重试
  4. 工具返回结果后，检查是否需要继续调用

教训：Agent 的循环检测和最大执行限制必须设置

---

案例 2：流式输出中断导致状态丢失

现象：用户正在下载文件，连接断开，任务状态丢失，用户不知道是否完成

根因：
  - 任务状态没有持久化
  - 服务重启后无法恢复

解决：
  1. 所有任务状态实时持久化
  2. 断连重连后能查询任务状态
  3. 支持从断点恢复

教训：Agent 的状态必须持久化，不能只存内存

---

案例 3：上下文溢出导致回答质量下降

现象：长会话后 Agent 开始给出前后矛盾的答案

根因：
  - 上下文超出窗口，早期关键信息被截断
  - 截断了用户早期的约束条件

解决：
  1. 智能截断算法，识别并保留关键信息
  2. 定期将重要信息摘要存入长期记忆
  3. 上下文超 80% 时预警

教训：上下文管理必须智能化，不能简单截断
```
