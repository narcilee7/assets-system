| 指标 | 目标 | 说明 |

## 性能瓶颈分析

### 瓶颈 1：密码验证（bcrypt）

#### 问题

bcrypt 计算慢（设计目标：防暴力破解），单次验证 10-50ms。

#### 优化方案

| 方案 | 实现 | 效果 |
|------|------|------|
| **缓存验证结果** | 5min 内同一用户不重复验证 | 降低 90% 验证次数 |
| **多实例水平扩展** | 多节点分担负载 | QPS 线性扩展 |
| **异步验证** | 验证码请求异步化 | 响应更快 |

#### 缓存策略

```go
// 5min 内同一用户不重复验证密码
cacheKey := "pwd:verified:" + userID
if cached := redis.Get(cacheKey); cached == "1" {
    return nil  // 已验证过
}

// 验证密码
err := bcrypt.CompareHashAndPassword(hash, password)
if err == nil {
    redis.Set(cacheKey, "1", 5*time.Minute)
}
```

---

### 瓶颈 2：Redis 访问（Token 存储）

#### 问题

Token 黑名单、Refresh Token 存储在 Redis，高 QPS 下 Redis 成为瓶颈。

#### 优化方案

| 方案 | 实现 | 效果 |
|------|------|------|
| **本地缓存** | 热点 Token 缓存本地 | Redis QPS 降低 80% |
| **Pipeline** | 批量操作 | 网络往返减少 |
| **读写分离** | 主从 Redis | 读取能力翻倍 |

## 容量规划

### QPS 容量估算

```
目标 QPS = 10W（Token 验证）

单节点 Token 验证 QPS = 10W（JWT 验签纯 CPU）
所需节点数 = 10W / 10W = 1 节点

考虑冗余（2 节点热备）：
  实际配置 = 2-3 节点
```

### Redis 容量估算

```
Token 黑名单：
  - 每天撤销 Token 数：10W
  - 每条约 100 bytes
  - TTL = 15min（平均）
  - 内存 ≈ 10W × 100 = 10MB

Refresh Token：
  - 在线用户数：100W
  - 每条约 200 bytes
  - 内存 ≈ 100W × 200 = 200MB

总计：约 1GB Redis 足够
```
