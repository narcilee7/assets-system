# Interview

## 核心追问与权衡讨论

---

### 1. Session 和 JWT 怎么选？

#### 讨论要点

```
Session：
  - 数据存服务端，Redis
  - 优点：可主动撤销
  - 缺点：需要共享 Session 存储
  - 适用：传统 Web 应用

JWT：
  - 数据自包含，存客户端
  - 优点：无状态，天然分布式
  - 缺点：不能主动撤销（除非黑名单）
  - 适用：API、微服务

选择建议：
  - 前后端分离 → JWT
  - 传统服务端渲染 → Session
  - 需要主动撤销 → Session 或短期 JWT + 黑名单
```

---

### 2. Refresh Token 如何轮换？

#### 讨论要点

```
轮换策略：

1. 不轮换
   - Refresh Token 长期有效
   - 风险：Token 被盗后长期可用

2. 滑动窗口
   - 每次使用后颁发新 Token
   - 旧 Token 作废
   - 推荐：安全体验平衡

3. 一次性
   - 使用后必须重新登录
   - 最安全但体验差
```

---

### 3. Token 泄漏后如何撤销？

#### 讨论要点

```
方案：

1. 短期 Token
   - Access Token 15min，过期即失效

2. Token 版本号
   - 用户修改密码 → 版本号递增
   - Token Payload 带版本号

3. 黑名单
   - Redis 存储被撤销的 Token JTI
   - 验证时检查黑名单
```

### 4. RBAC 和 ABAC 怎么建模？

#### 讨论要点

```
RBAC（基于角色）：
  - User → Role → Permission
  - 简单，适合粗粒度

ABAC（基于属性）：
  - 资源属性 + 用户属性 动态决策
  - 复杂，适合细粒度

混合：
  - RBAC 做角色判断
  - ABAC 做数据归属判断
```

### 5. 多租户权限如何隔离？

#### 讨论要点

```
隔离方案：

1. Tenant ID 字段隔离
   - 所有表加 tenant_id
   - 查询时自动加 tenant_id 条件

2. 独立 Schema
   - 每个租户独立 Schema
   - 适合租户数少的场景

3. 行级安全（RLS）
   - 数据库层强制隔离
```

### 6. 生产环境踩过的坑

#### 案例 1：JWT 泄漏无法撤销

用户 Token 被盗，无法主动撤销。

教训：短期 Token + Token 版本号 + 黑名单。

#### 案例 2：密码找回绕过

密码重置链接可被猜测遍历。

教训：密码重置 Token 必须一次性使用，用后即焚。

#### 案例 3：RBAC 权限过大

管理员角色权限过大，越权操作。

教训：RBAC + ABAC 混合，敏感操作二次验证。
