# Problem

## 需求

构建一个生产级的消息队列系统，实现异步任务分发、消息可靠传递、重试机制、顺序保证和消费者幂等性，在高吞吐、海量消息场景下保证系统稳定。

### 功能需求

#### 1. 消息传递语义（Delivery Semantics）

| 语义 | 描述 | 场景 |
|------|------|------|
| **At-most-once** | 消息最多传递一次，可能丢失 | 日志收集、监控指标 |
| **At-least-once** | 消息至少传递一次，不会丢失，但可能重复 | 大多数业务场景 |
| **Exactly-once** | 消息恰好传递一次，既不丢失也不重复 | 支付、计费 |

#### 2. 消息持久化（Persistence）

| 方案 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| **内存队列** | 消息存内存 | 超低延迟 | 断电丢失 |
| **磁盘持久化** | 消息写磁盘 | 可靠 | 有写入延迟 |
| **内存+磁盘** | 热数据内存，冷数据磁盘 | 平衡 | 实现复杂 |

#### 3. ACK 机制（Acknowledgement）

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| **自动 ACK** | 消息投递后立即 ACK | 允许丢失的消息 |
| **手动 ACK** | 消费者处理完成后手动 ACK | 重要消息 |
| **批量 ACK** | 累积 N 条后批量 ACK | 提升吞吐 |
| ** negativa ACK（NACK）** | 处理失败后 NACK，触发重试 | 需要重试的场景 |

#### 4. 死信队列（Dead Letter Queue / DLQ）

处理失败且超过重试次数的消息进入 DLQ。

| 功能 | 描述 |
|------|------|
| **DLQ 路由** | 重试 N 次后自动进入 DLQ |
| **DLQ 监控** | DLQ 积压告警 |
| **DLQ 消费** | 支持人工处理 DLQ 消息 |
| **DLQ 重新入队** | 修复后重新投放到原队列 |

#### 5. 消息重试（Retry）

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| **立即重试** | 失败后立即重试 | 瞬时故障 |
| **延迟重试** | 等待固定时间后重试 | 临时故障 |
| **指数退避重试** | 延迟时间指数增长 | 避免重试风暴 |
| **最大重试次数** | 超过后进入 DLQ | 避免无限重试 |

#### 6. 顺序保证（Ordering）

| 级别 | 描述 | 实现难度 |
|------|------|----------|
| **全局顺序** | 所有消息严格按顺序 | 难 |
| **分区顺序** | 同一 Partition 内有序 | 中 |
| **无顺序** | 不保证顺序 | 易 |

#### 7. 消费者幂等（Idempotency）

| 方案 | 描述 |
|------|------|
| **消息去重表** | 根据 message_id 去重 |
| **业务幂等** | 业务层实现（如订单状态机） |
| **唯一键约束** | 数据库唯一键防止重复 |

### 非功能需求

| 指标 | 目标 | 说明 |
|------|------|------|
| 吞吐量 | > 10W QPS | 单队列 |
| 消息延迟 P99 | < 10ms | 端到端 |
| 消息持久性 | 99.9999% | 消息不丢失 |
| 系统可用性 | 99.99% | 队列服务本身 |
| 消息堆积容量 | > 1000W | 消费者故障时堆积 |

### 约束

- **消息不可变**：消息一旦发送，不能修改
- **消息 ID 唯一**：每条消息有全局唯一 ID
- **消费者独立**：各消费者独立消费，互不影响
- **水平扩展**：支持队列和消费者的水平扩展
- **不保证跨分区顺序**：只有同分区消息有序

### 关键设计决策

#### 决策 1：消息持久化策略

| 策略 | 写入路径 | 性能 | 可靠性 |
|------|----------|------|---------|
| **异步刷盘** | 写内存 + 定期刷盘 | 最高 | 中（可能丢数秒数据）|
| **同步刷盘** | 写内存 + 同步刷盘 | 中 | 高 |
| **Write-Ahead Log** | 追加写 WAL + 内存 | 高 + 可靠 | 最高 |

**推荐**：**异步刷盘 + WAL**，平衡性能和可靠性。

#### 决策 2：消息传递语义选择

| 场景 | 推荐语义 | 原因 |
|------|----------|------|
| **支付/计费** | Exactly-once | 不能重复扣款 |
| **订单通知** | At-least-once | 不能丢失通知 |
| **日志收集** | At-most-once | 允许丢日志 |
| **实时监控** | At-most-once | 允许丢指标 |

**推荐**：**At-least-once + 幂等** 作为默认（大多数业务场景）。

#### 决策 3：分区策略（Partition /Shard）

| 策略 | Key 选择 | 适用场景 |
|------|----------|-----------|
| **用户 ID 分区** | hash(user_id) % partitions | 同一用户消息有序 |
| **订单 ID 分区** | hash(order_id) % partitions | 同一订单操作有序 |
| **随机分区** | round_robin | 无顺序要求 |

**推荐**：**按业务 ID 分区**，保证同实体消息有序。

#### 决策 4：消费者组设计

```
Consumer Group 概念：
  - 多个消费者组成一个消费组
  - 消息在组内只被一个消费者消费
  - 不同消费组可以各自独立消费同一条消息

设计原则：
  - 同一消费组内：竞争消费，提高吞吐
  - 不同消费组间：广播消费，各司其职
  - 消费者数量 ≤ Partition 数量
```

### 真实案例与踩坑

#### 案例 1：重复消费导致订单重复扣款

用户支付后，系统发送消息到 MQ，消费者收到消息后执行业务（扣款），但 ACK 失败导致重复消费，用户被扣了两次钱。

**教训**：
- 消费者必须实现幂等（如订单状态机：只有待支付才能扣款）
- 消息 ID 去重表：收到消息先去重再处理
- 业务层幂等是最后防线

#### 案例 2：重试风暴导致系统雪崩

消费者失败后立即重试，导致大量重试请求同时打到下游服务，下游服务被打垮。

**教训**：
- 重试必须用指数退避（1s → 2s → 4s）
- 设置最大重试次数限制
- 使用熔断器保护下游

#### 案例 3：消费者慢导致消息堆积

消费者处理一条消息需要 10 秒，消息堆积 100W 条，新消息延迟高达 1000 秒。

**教训**：
- 消费者必须优化单条消息处理时间
- 快速失败：处理超时要立即 NACK
- 消费者需要水平扩展（增加实例）
- 监控消费延迟，及时扩容
