# Interview

## 核心追问与权衡讨论

---

### 1. at-most-once、at-least-once、exactly-once 怎么理解？

#### 面试官想知道

- 你是否理解三种传递语义的本质
- 你是否能根据场景选择合适的语义
- 你是否理解实现复杂度

#### 讨论要点

```
三种语义对比：

1. at-most-once（最多一次）
   - 含义：消息可能丢失，但不会重复
   - 实现：Producer 发送后不等待 ACK
   - 场景：日志收集、监控指标

2. at-least-once（至少一次）
   - 含义：消息不会丢失，但可能重复
   - 实现：Producer 等待 ACK，失败重试
   - 场景：大多数业务场景

3. exactly-once（恰好一次）
   - 含义：消息既不丢失也不重复
   - 实现：at-least-once + 幂等消费者
   - 场景：支付、计费等关键业务

权衡：
  - 越强的语义，实现越复杂，成本越高
  - 根据业务场景选择合适的语义
```

### 2. ACK 什么时候提交？

#### 面试官想知道

- 你是否理解 ACK 时机的重要性
- 你是否能设计合理的 ACK 策略
- 你是否理解 ACK 失败的处理

#### 讨论要点

```
ACK 时机选择：

1. 消息投递后立即 ACK（自动 ACK）
   - 时机：消息到达消费者队列后
   - 优点：吞吐高，延迟低
   - 缺点：可能丢消息（消费者处理失败但已 ACK）
   - 场景：不重要消息

2. 业务处理完成后 ACK（手动 ACK）
   - 时机：业务逻辑执行完成
   - 优点：不丢消息
   - 缺点：处理慢导致消息堆积
   - 场景：重要消息

3. 批量 ACK
   - 时机：累积 N 条后批量 ACK
   - 优点：减少 ACK 次数，提升吞吐
   - 缺点：失败时重试粒度大
   - 场景：高吞吐场景

最佳实践：
  - 重要消息：手动 ACK
  - 高吞吐场景：批量 ACK + 快速失败
```

### 3. 失败重试如何避免放大故障？

#### 面试官想知道

- 你是否理解重试风暴的危害
- 你是否有系统的重试策略设计
- 你是否理解指数退避和熔断

#### 讨论要点

```
重试风暴问题：

场景：下游服务故障，所有请求同时重试
       1000 个消息重试 → 10000 个请求 → 下游更慢 → 更多超时 → 更多重试

解决方案：

1. 指数退避
   - 重试延迟：1s → 2s → 4s → 8s
   - 加随机 jitter 避免所有请求同时重试

2. 熔断器
   - 失败次数超过阈值 → 熔断
   - 熔断期间直接失败，不重试
   - 等待一段时间后放行探测

3. 限流保护
   - 下游限流
   - 上游降级

4. 最大重试次数
   - 超过后进入 DLQ
   - 人工处理
```

### 4. 消费者幂等怎么做？

#### 面试官想知道

- 你是否理解幂等性的重要性
- 你是否有多种幂等实现方案
- 你是否能处理边界情况

#### 讨论要点

```
幂等方案：

1. 消息去重表
   - 收到消息 → 检查 dedup 表
   - 存在 → 跳过
   - 不存在 → 处理 → 标记为已处理

2. 业务状态机
   - 订单状态机：pending → paid
   - 只有 pending 才能支付
   - 已支付直接返回成功

3. 唯一键约束
   - 业务表有唯一键
   - 重复插入报错

4. 版本号控制
   - 消息带版本号
   - 处理前检查版本

最佳实践：
  - 消息去重表（基础）
  - 业务状态机（兜底）
```

### 5. 如何处理消息积压？

#### 面试官想知道

- 你是否理解积压的原因
- 你是否有系统的扩容方案
- 你是否能设计背压机制

#### 讨论要点

```
积压原因：

1. 消费者处理慢
   - 业务逻辑复杂
   - 下游调用慢

2. 消费者数量不足
   - 消费者崩溃
   - 消费者实例不足

3. 下游故障
   - 下游服务不可用
   - 消费者快速失败但重试堆积

解决方案：

1. 扩容消费者
   - 增加消费者实例
   - 重新分配 Partition

2. 优化消费者
   - 快速失败
   - 异步处理非关键逻辑

3. 背压机制
   - 生产端限流
   - 队列积压时减缓发送

4. 消费者分级
   - 重要消息优先消费
   - 非重要消息丢弃
```

### 6. 如何保证消息顺序？

#### 面试官想知道

- 你是否理解顺序保证的代价
- 你是否能设计有序消息方案
- 你是否理解乱序的场景和处理

#### 讨论要点

```
顺序保证：

1. 全局有序
   - 只有一个 Partition
   - 缺点：吞吐低
   - 场景：金融交易

2. 分区有序
   - 同一 Key 的消息发送到同一 Partition
   - 缺点：热点 Key 导致 Partition 不均
   - 场景：大多数业务

3. 同实体有序
   - 同一用户的消息有序
   - 按 user_id 分区

乱序处理：

1. 时间戳排序
   - 消费者按消息时间戳排序

2. 序列号机制
   - 消息带 sequence
   - 消费者按序列号顺序处理

3. 乱序检测
   - 监控乱序率
   - 乱序时告警
```

### 7. 生产环境踩过的坑

#### 案例 1：重复消费导致订单重复扣款

现象：消费者处理失败后重试，用户被扣了两次钱。

根因：没有幂等，重复消息被重复处理。

解决：
- 业务状态机：只有 pending 才能扣款
- 消息去重表：重复消息跳过

教训：幂等是消费者的必备能力。

---

#### 案例 2：重试风暴导致下游服务雪崩

现象：下游服务抖动，所有消息同时重试，下游被打垮。

根因：没有指数退避，所有重试同时发出。

解决：
- 指数退避 + 随机 jitter
- 熔断器保护
- 最大重试次数限制

教训：重试必须有退避策略。

---

#### 案例 3：消费者重启导致消息丢失

现象：消费者处理中时崩溃，消息未 ACK，但已被投递。

根因：消息投递后立即 ACK，但业务未处理完。

解决：
- 业务处理完成后才 ACK
- 使用事务机制（消息处理和状态更新原子）

教训：ACK 时机要等业务处理完成。
