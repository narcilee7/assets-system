# Problem

## 需求

构建一个生产级的大文件上传系统，支持分片上传、秒传、断点续传、合并，在高并发、海量文件场景下保证可靠性和一致性。

### 功能需求

#### 1. 分片上传（Chunked Upload）

将大文件拆分为多个小块（chunk），逐个上传。

| 功能 | 描述 |
|------|------|
| **固定分片大小** | 每片 5MB（可配置），支持任意大小文件 |
| **动态分片大小** | 根据文件大小自动调整（如大文件用更大的片） |
| **并行上传** | 同时上传多个分片，提升速度 |
| **并发控制** | 限制同时上传的分片数（避免占满带宽）|

#### 2. 秒传（Instant Upload / Deduplication）

通过文件 Hash 判断文件是否已存在，已存在则直接复用，无需真正上传。

| 步骤 | 描述 |
|------|------|
| 1. 计算文件 Hash | 前端计算文件 MD5/SHA256 |
| 2. 查询后端 | 检查该 Hash 是否已存在 |
| 3. 存在则复用 | 返回已有文件的 ID，标记秒传成功 |
| 4. 不存在则分片上传 | 正常走分片上传流程 |

#### 3. 断点续传（Resumable Upload）

上传中断后（如网络断开、页面刷新），能从断点恢复，不需要重新开始。

| 功能 | 描述 |
|------|------|
| **查询已上传分片** | 从后端获取已上传的分片列表 |
| **跳过已上传分片** | 只上传缺失的分片 |
| **进度持久化** | 上传进度存 localStorage，刷新后可恢复 |
| **服务端状态存储** | 分片上传状态存 Redis/DB |

#### 4. 文件合并（Merge）

所有分片上传完成后，请求服务端合并为完整文件。

| 步骤 | 描述 |
|------|------|
| 1. 检查分片完整性 | 验证所有分片已上传 |
| 2. 原子合并 | 按顺序拼接分片 |
| 3. 校验文件 Hash | 合并后校验整体 Hash |
| 4. 清理分片 | 删除临时分片文件 |

#### 5. 文件 Hash 计算

| 方法 | 优点 | 缺点 |
|------|------|------|
| **MD5** | 计算快，兼容性好 | 有哈希碰撞风险（极低）|
| **SHA-256** | 安全性更高 | 计算慢 30% |
| **CRC32** | 计算最快 | 碰撞风险较高 |
| **Web Crypto API** | 原生浏览器 API，可计算大文件 | 支持度稍差 |

#### 6. 存储方案

| 方案 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| **本地磁盘** | 小文件、测试环境 | 简单 | 难以扩展 |
| **NFS / 分布式文件系统** | 中小规模 | 共享存储 | 性能一般 |
| **对象存储（OSS/S3）** | 大规模、海量文件 | 弹性扩展、高可用 | 需要额外费用 |

### 非功能需求

| 指标 | 目标 | 说明 |
|------|------|------|
| 单文件大小 | 100GB+ | 支持超大文件 |
| 分片大小 | 5MB（默认，可配置）| 平衡重试开销和并发效率 |
| 上传速度 | 取决于用户带宽 | 分片并行加速 |
| 秒传成功率 | > 30% | 热门文件复用率高 |
| 断点恢复时间 | < 1s | 刷新页面后快速恢复 |
| 合并成功率 | > 99.9% | 含 Hash 校验 |
| 上传可用性 | 99.99% | 上传服务本身 |

### 约束

- **浏览器内存限制**：不能一次性读取整个文件到内存，必须流式读取
- **网络超时**：单个分片上传有超时限制，断网需重试
- **Hash 计算瓶颈**：大文件（10GB+）的 MD5 计算在主线程会导致 UI 卡顿
- **分片顺序**：合并时必须按顺序拼接，否则文件损坏
- **幂等性**：重复的上传请求不能创建重复的分片或合并
- **存储清理**：上传中断的临时分片需要清理机制

### 关键设计决策

#### 决策 1：分片大小如何选择？

| 分片大小 | 优点 | 缺点 | 适用场景 |
|----------|------|------|----------|
| 1MB | 重试代价小 | 分片数多，管理复杂 | 网络不稳定 |
| 5MB | 平衡点 | - | 默认推荐 |
| 10MB | 分片少，HTTP 开销小 | 重试代价大 | 网络稳定 |
| 动态调整 | 根据文件大小自动选择 | 实现复杂 | 追求最优 |

**推荐**：**5MB 默认 + 可配置**，绝大多数场景最优。

#### 决策 2：秒传的 Hash 用 MD5 还是 SHA-256？

| Hash | 碰撞风险 | 计算速度 | 适用场景 |
|------|----------|----------|----------|
| MD5 | 极低（实际可忽略）| 快 30% | 一般场景 |
| SHA-256 | 几乎为零 | 稍慢 | 安全敏感（如代码托管）|

**推荐**：**MD5 作为默认**（速度快，碰撞风险实际可忽略），支持 SHA-256 选项。

#### 决策 3：上传状态存在哪里？

| 方案 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| **localStorage** | 前端持久化 | 刷新页面可恢复 | 有大小限制（5MB），跨浏览器不共享 |
| **Redis** | 服务端存储 | 可靠，支持分布式 | 增加服务复杂度 |
| **数据库** | MySQL/PostgreSQL | 持久化，支持查询 | 性能一般 |
| **混合** | localStorage + 服务端兜底 | 兼顾体验和可靠 | 实现复杂 |

**推荐**：**前端 localStorage + 服务端 Redis**（分片状态在 Redis，上传进度在 localStorage）。

#### 决策 4：分片并发数如何控制？

| 策略 | 实现 | 效果 |
|------|------|------|
| **固定并发** | 同时上传最多 N 个分片 | 简单可控 |
| **动态并发** | 根据带宽动态调整 | 充分利用带宽 |
| **自适应** | 慢分片降低并发 | 防止慢的分片拖后腿 |

**推荐**：**固定并发 3-5**（平衡速度和资源占用）。

### 真实案例与踩坑

#### 案例 1：合并接口重复调用导致文件重复

用户上传完成后，连续点击"完成"按钮两次，导致合并接口被调用两次，触发了并发合并，产生了两个不完整的文件。

**教训**：
- 合并接口必须幂等（重复调用返回相同结果）
- 合并前检查分片完整性
- 合并状态机：pending → merging → completed

#### 案例 2：超大文件 MD5 计算导致浏览器卡死

用户选择一个 50GB 的视频文件，前端开始计算 MD5，由于在主线程计算，浏览器 UI 完全卡死，用户以为程序崩溃。

**教训**：
- 大文件 MD5 必须用 Web Worker 计算
- 分片计算 Hash（每片单独计算，最后合并）
- 提供计算进度，让用户知道程序在运行

#### 案例 3：断点续传后文件 Hash 不一致

用户上传一个 10GB 文件，进度 80% 时断网，重新连接后断点续传，但合并后文件 Hash 和原始文件 Hash 不一致。

**根因**：断点续传时，分片合并顺序错误或某个分片数据损坏。

**教训**：
- 合并时必须校验每个分片的 Hash
- 合并后校验整体 Hash
- 分片要有序号，合并严格按序号排序
