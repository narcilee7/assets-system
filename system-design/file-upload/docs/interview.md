## 核心追问与权衡讨论

---

### 1. 如果文件有 10GB，浏览器端 MD5 怎么优化？

#### 面试官想知道

- 你是否理解大文件 Hash 计算的性能问题
- 你是否有实际的前端性能优化经验
- 你是否理解 Web Worker 的使用场景

#### 讨论要点

```
问题分析：
  - 10GB 文件一次性读入内存 = 内存爆炸
  - 主线程计算 = UI 卡死
  - MD5 计算本身也需要时间

优化方案：

1. Web Worker 后台计算
   - 不阻塞主线程
   - 用户可以继续操作
   - 可显示计算进度

2. 分片计算 Hash
   - 分片读取（5MB 一片）
   - 每片单独计算
   - 合并结果
   - 可以显示进度百分比

3. 使用 Web Crypto API
   - 原生浏览器 API
   - 可处理大文件（流式读取）
   - SHA-256 支持好

4. 增量计算
   - 只在必要时计算（如秒传检查）
   - 小文件直接计算
   - 大文件分片计算
```

#### 追问

**Q：MD5 和 SHA-256 在浏览器端性能差异大吗？**

A：
```
实测差异：
  - SHA-256 比 MD5 慢约 30%
  - 但 SHA-256 安全性更高

选择建议：
  - 一般场景：MD5 足够（碰撞概率极低）
  - 安全敏感场景（如代码托管）：SHA-256
  - 可以让客户端选择
```

### 2. 如果合并接口失败，怎么保证最终一致？

#### 面试官想知道

- 你是否理解分布式系统的最终一致性
- 你是否有幂等性设计经验
- 你是否能处理失败后的恢复流程

#### 讨论要点

```
合并失败场景：

1. 网络超时
   - 不确定服务端是否收到请求
   - 可能是成功了但返回失败

2. 服务端错误
   - 分片数据可能已损坏
   - 需要重新上传

3. Hash 不一致
   - 合并后校验失败
   - 需要删除已合并的文件

幂等性设计：

1. 接口幂等
   - 重复调用返回相同结果
   - 合并前检查状态

2. 状态机保护
   - initialized → uploading → merging → completed
   - 只有在正确状态才能转换

3. 分布式锁
   - 同一上传任务同时只能有一个合并请求
   - 用 Redis 锁保护

4. 最终一致性
   - 定时扫描失败任务
   - 自动重试或人工介入
```

#### 追问

**Q：如何检测和处理"假失败"（实际成功但返回失败）？**

A：
```
场景：网络超时，但服务端实际已合并成功

检测方法：
  1. 查询任务状态（GET /upload/{id}/status）
  2. 如果状态是 completed，说明实际成功了

处理流程：
  1. 收到超时错误
  2. 不立即重试
  3. 先查询状态
  4. 根据状态决定下一步
```

### 3. 如果多个用户同时上传同一个文件，怎么去重？

#### 面试官想知道

- 你是否理解秒传的实现原理
- 你是否能设计高效的去重机制
- 你是否理解 Hash 碰撞的处理

#### 讨论要点

```
去重策略：

1. 秒传（Instant Upload）
   - 前端计算文件 Hash
   - 查询后端是否存在
   - 存在则直接返回，不真正上传

2. 秒传 Hash 选择
   - MD5：计算快，碰撞率极低
   - SHA-256：更安全
   - 可以用组合：MD5 + 文件大小

3. 秒传后校验
   - 秒传返回后，随机抽样校验
   - 下载文件片段验证 Hash
   - 确保不是 Hash 碰撞导致的错误

4. 并发上传同一个文件
   - 用户 A 和用户 B 同时上传相同文件
   - 秒传检查都通过
   - 各自返回自己的 file_id
   - 实际上指向同一个存储对象（OSS 内部去重）
```

#### 追问

**Q：Hash 碰撞了怎么办？**

A：
```
Hash 碰撞概率：
  - MD5：2^128 分之一，实际可忽略
  - SHA-256：更不可能

实际处理：
  1. 碰撞概率极低，不做特殊处理
  2. 秒传后抽样校验
  3. 如果发现碰撞，当作普通失败处理

如果担心碰撞：
  - 用 MD5 + 文件大小 组合
  - 或用 MD5 + SHA-256 组合
  - 碰撞概率进一步降低
```

### 4. 如果上传过程中刷新页面，如何恢复？

#### 面试官想知道

- 你是否有状态持久化经验
- 你是否理解断点续传的设计
- 你是否能处理各种中断场景

#### 讨论要点

```
恢复流程：

1. 页面刷新前
   - 定期保存进度到 localStorage
   - 保存：upload_id、已上传分片列表、进度百分比

2. 页面刷新后
   - 从 localStorage 读取保存的进度
   - 检查是否过期（24h 内）
   - 查询服务端已上传的分片
   - 计算缺失的分片
   - 只上传缺失的分片

3. localStorage 数据结构
   {
     upload_id: "upload-abc",
     file_name: "video.mp4",
     file_hash: "md5:...",
     total_chunks: 2048,
     uploaded_chunks: [0, 1, 2, ..., 99],
     progress_percent: 4.88,
     last_updated: 1717200000
   }

4. 服务端状态
   - 分片状态存在 Redis
   - 刷新后从 Redis 恢复
```

#### 追问

**Q：如果 localStorage 也被清空了怎么办？**

A：
```
场景：用户清除了浏览器数据，或者换浏览器

服务端兜底：
  1. 分片状态在 Redis 有备份
  2. 用户可以从服务端查询已上传的分片
  3. 只需要重新输入 upload_id

用户体验：
  1. 提示用户复制 upload_id
  2. 或者让用户重新选择文件（用 file_name + file_size + file_hash 识别）
```

### 5. 分片大小如何选择？

#### 面试官想知道

- 你是否理解分片大小的权衡
- 你是否能根据场景选择最优分片大小
- 你是否理解分片数和重试代价的关系

#### 讨论要点

```
分片大小权衡：

分片太大（50MB）：
  优点：HTTP 请求数少，协议开销小
  缺点：重试代价大（50MB 重新上传）

分片太小（1MB）：
  优点：重试代价小
  缺点：请求数多，协议开销大

选择建议：
  - 默认 5MB（平衡点）
  - 网络不稳定：2-3MB
  - 网络稳定大文件：10MB

动态调整：
  - 小文件（< 10MB）：不分片
  - 中文件（10MB-1GB）：5MB
  - 大文件（> 1GB）：10-20MB
```

### 6. 如何保证分片合并顺序正确？

#### 面试官想知道

- 你是否理解文件合并的正确性要求
- 你是否有数据完整性校验的经验
- 你是否能设计可靠的合并流程

#### 讨论要点

```
合并顺序问题：

1. 分片编号保证
   - 每个分片有 index（0, 1, 2, ...）
   - 上传时记录 index
   - 合并时严格按 index 排序

2. 数据库唯一索引
   - UNIQUE KEY (task_id, chunk_index)
   - 防止重复上传同一分片

3. 合并前完整性检查
   - 检查所有分片是否已上传
   - 检查分片数量是否正确
   - 检查每个分片的 Hash

4. 合并后 Hash 校验
   - 计算合并后文件的 Hash
   - 对比客户端提供的 Hash
   - 不一致则报错
```

### 7. 生产环境踩过的坑

#### 面试官想知道

- 你是否有真实的故障处理经验
- 你是否能从故障中总结系统性改进

#### 典型案例

```
案例 1：合并接口重复调用导致数据错乱

现象：用户快速双击"完成"按钮，触发了两次合并请求
       服务端创建了两个合并任务，文件被合并了两次

根因：
  - 合并接口没有幂等性保护
  - 没有状态机检查
  - 没有分布式锁

解决：
  1. 合并接口幂等：重复调用返回相同结果
  2. 状态机保护：只有 uploading 状态才能进入 merging
  3. 分布式锁：Redis SETNX 保护
  4. 前端：防重复点击，合并按钮点击后禁用

教训：所有写接口必须幂等，不能依赖前端防重

---

案例 2：10GB 文件 MD5 计算导致浏览器崩溃

现象：用户选择 10GB 视频文件，浏览器卡死 30 秒后崩溃

根因：
  - MD5 计算在主线程
  - 10GB 文件一次性读入内存
  - 内存占用暴增导致 OOM

解决：
  1. Web Worker 后台计算
  2. 分片计算（每片 5MB）
  3. 提供计算进度
  4. 大文件提示用户等待

教训：大文件处理必须考虑前端性能，不能阻塞主线程

---

案例 3：秒传 Hash 不一致导致文件损坏

现象：秒传成功后，文件下载后无法播放

根因：
  - 秒传只检查 Hash，没有校验内容
  - 两个不同文件 MD5 恰好相同（Hash 碰撞）
  - 用户拿到了别人的文件

解决：
  1. 秒传后抽样校验（下载片段验证）
  2. 高安全场景用 MD5 + 文件大小组合
  3. 重要文件支持 SHA-256

教训：秒传不能完全依赖 Hash，要有一定校验机制
```
