# Interview

## 经典追问

### Q1: 令牌桶和漏桶的本质区别？

**令牌桶**：以恒定速率产生令牌，请求消耗令牌，允许突发（桶里有存货就能一次拿走）。

**漏桶**：以恒定速率漏出，请求入桶，桶满则拒，强制平滑（无法突发）。

**选择**：
- 需要容忍合理突发 → 令牌桶（绝大多数 API 场景）
- 需要严格匀速输出 → 漏桶（带宽限制、下游保护）

### Q2: 滑动窗口比固定窗口好在哪里？代价是什么？

**好**：避免窗口边界突发（如 00:59 和 01:00 连续打满两个窗口）。

**代价**：
- 需要存储多个子窗口的计数（内存/Redis 开销增大）
- 计算时需要聚合，延迟略高
- 分布式实现需要 Lua 脚本保证原子性

**折中方案**：滑动日志（记录每个请求时间戳）精确但内存爆炸；实际常用 **滑动窗口日志的近似版**（分 10 个子窗口）。

### Q3: 如何实现按优先级放行？

**方案 A：独立配额池**

```
普通用户：pool_normal，limit=100
VIP 用户：pool_vip，limit=1000
```

判定前查用户等级，路由到不同池。

**方案 B：优先级队列（漏桶场景）**

高优先级请求插入队首，低优先级插入队尾。

**方案 C：抢占配额**

VIP 可临时借用普通池的剩余配额。

### Q4: 百万 QPS 限流判断如何优化到亚毫秒？

1. **本地缓存为主**：99% 判断不走网络
2. **无锁原子操作**：atomic.AddInt64 替代 Mutex
3. **批量预取**：一次 Redis 交互换 100 次本地判断
4. **CPU 亲和性**：限流线程绑定特定核心，避免缓存失效
5. **SIMD 批量判断**：对同 resource 的批量请求做向量化处理

### Q5: 限流和熔断的区别与协同？

| | 限流（Rate Limit） | 熔断（Circuit Breaker） |
|--|-------------------|------------------------|
| 控制对象 | 请求速率 | 错误率/响应时间 |
| 触发条件 | 超过预设阈值 | 下游服务异常 |
| 恢复策略 | 时间窗口到期自动恢复 | 半开探测成功后恢复 |
| 粒度 | 细（用户/接口/IP） | 粗（服务/接口） |

**协同**：网关层先做限流（防刷），再做熔断（保护后端）。

```
Request → Rate Limit → Circuit Breaker → Backend
```

### Q6: 分布式限流允许多少误差？如何做到严格精确？

**误差来源**：
- Redis 网络延迟：请求在途时计数已变
- 时钟漂移：窗口边界不一致
- 本地缓存：异步同步的延迟

**严格精确的方案**（代价高）：
- 中央计数器 + 同步 RPC（每请求都确认）→ 延迟爆炸
- 分布式共识（Paxos/Raft 维护计数）→ 吞吐量爆炸

**结论**：限流本身允许近似，追求严格精确是过度设计。

## 设计权衡总结

| 维度 | 选项 A | 选项 B | 推荐 |
|------|--------|--------|------|
| 算法 | 令牌桶 | 滑动窗口 | 令牌桶（通用）+ 滑动窗口（防突） |
| 存储 | Redis | 本地内存 | 本地为主 + Redis 兜底 |
| 一致性 | 强一致 | 最终一致 | 最终一致（限流允许误差） |
| 故障 | Fail-open | Fail-close | 接口级配置 |
| 部署 | 中心化 | 去中心化 | 各节点自治 + 配置同步 |
