# Failure Mode

## F1: Redis 不可用 / 网络抖动

| 场景 | 影响 | 应对 |
|------|------|------|
| Redis 短暂超时 | 分布式限流失效 | 降级为**单机限流**或 **fail-open** |
| Redis 长期不可用 | 限流退化为各节点自治 | 开启**本地严格模式**（单机配额 = 分布式配额 / 节点数） |
| Redis 部分分片故障 | 部分 key 限流失效 | 一致性 Hash 重定向到备用 Redis |

**策略选择**：
- **Fail-open**：限流组件故障不影响主业务，但可能打垮后端（适合非核心接口）
- **Fail-close**：限流组件故障时严格限流，可能影响可用性（适合核心保护场景）
- **混合策略**：根据接口重要性动态选择

## F2: 规则配置错误

| 场景 | 影响 | 应对 |
|------|------|------|
| 规则 limit 设为 0 | 全部请求被拒 | 配置校验：limit > 0；发布前灰度 |
| 规则维度组合爆炸 | key 数量激增，Redis 内存耗尽 | 维度组合数上限保护；异常维度兜底 |
| 规则循环依赖 | 无 | 规则本身无依赖关系 |

## F3: 热点 Key

| 场景 | 影响 | 应对 |
|------|------|------|
| 全局限流 key 成为热点 | Redis 单节点 CPU 打满 | 本地缓存 + 异步同步；Redis Cluster 分片 |
| 特定用户被集中限流 | 该用户的 key 高频访问 | 本地缓存该用户配额，减少 Redis 访问 |

**热点 Key 优化**：

```
本地缓存层（L1）
  ├── 命中且未过期 ──► 直接返回
  ├── 命中但即将过期 ──► 异步刷新（避免惊群）
  └── 未命中 ──► 查 Redis（L2）
```

## F4: 时钟漂移

| 场景 | 影响 | 应对 |
|------|------|------|
| 节点间时钟不同步 | 窗口边界判断不一致 | NTP 同步 + 容忍 ±1s 误差 |
| 时钟回拨 | 令牌桶 refill 异常 | 记录上次时间，回拨时冻结 refill |

```go
func (b *TokenBucket) safeRefill(now int64) {
    if now < b.lastRefill {
        // 时钟回拨，跳过本次 refill
        return
    }
    // ... normal refill
}
```

## F5: 惊群效应（Thundering Herd）

| 场景 | 影响 | 应对 |
|------|------|------|
| 缓存失效后大量请求同时查 Redis | Redis 瞬时压力激增 | 随机 jitter 过期时间；单线程异步刷新 |

## F6: 级联限流

| 场景 | 影响 | 应对 |
|------|------|------|
| A 服务限流 B，B 限流 C | 延迟累积，误杀正常请求 | 限流仅在入口层做，内部服务用熔断而非限流 |
