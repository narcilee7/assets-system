# Problem

## 需求

构建一个支持多粒度、多算法、可动态配置的限流系统，保护后端服务免受流量突增和恶意刷量的影响。

### 功能需求

1. **限流判断**：对请求进行快速通过/拒绝判定
2. **多算法支持**：令牌桶、滑动窗口、固定窗口、漏桶
3. **多粒度配置**：全局、用户、IP、接口、组合维度
4. **动态规则**：支持运行时热更新限流规则，无需重启
5. **降级策略**：限流触发后返回统一响应（429 / 自定义降级内容）
6. **优先级放行**：支持 VIP / 白名单绕过或独立配额
7. **批量查询**：支持批量接口的聚合限流判断

### 非功能需求

| 指标 | 目标 |
|------|------|
| 判断延迟 | P99 < 1ms（单机），P99 < 5ms（分布式） |
| 吞吐量 | 单机 10W+ QPS，集群 100W+ QPS |
| 一致性 | 分布式场景允许 ±1 个窗口的计数误差 |
| 可用性 | 限流组件故障不应影响主链路（fail-open） |
| 扩展性 | 规则数支持万级，不显著增加判断延迟 |

### 约束

- 不保证严格精确计数，允许可接受的近似
- 分布式限流依赖 Redis / 类似存储，需考虑网络抖动
- 时间以秒级窗口为主，支持毫秒级细粒度
