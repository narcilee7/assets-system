# Outbox Pattern Review

## 设计评估

### 优点

1. **原子性保证**：业务数据和 Outbox 记录在同一事务写入，确保要么同时成功，要么同时失败
2. **接口抽象良好**：Store 和 Queue 接口抽象，方便替换实现（InMemory → PostgreSQL / RabbitMQ）
3. **幂等设计**：事件 ID 基于 UUID，消费者可去重
4. **可观测性**：Worker 提供 poll 状态，可集成监控
5. **批量处理**：支持批量发布和批量标记，提高吞吐

### 可改进点

1. **消费者 cleanup 实现不完整**：当前 `EventConsumer.cleanup()` 是空实现，实际需要记录每个事件的时间戳才能按时间清理去重记录
2. **事务边界不够明确**：当前 `writeInTransaction` 是模拟实现，实际需要调用方控制数据库事务。建议文档说明如何与真实 ORM/DB 集成
3. **死信队列**：超过 maxRetries 的事件应该移到死信队列，而不是留在 outbox 表
4. **批量发布原子性**：当前逐个发布，如果中间失败会导致部分发布成功、部分失败。应该用事务性消息或重试机制
5. **Worker 位置追踪**：如果 Worker 在发布中途崩溃，重启后可能重复发布。建议用游标或版本号追踪

### 关键设计决策

| 决策 | 选择 | 理由 |
| --- | --- | --- |
| 事件 ID 生成 | UUID | 分布式唯一，无需中心化 |
| 存储 | Outbox 表 | 结构简单，查询高效 |
| 发布时机 | 后台轮询 | 实现简单，比 CDC 轻量 |
| 去重策略 | 消费者侧 | Outbox 只保证 at-least-once |
| 清理策略 | 时间窗口 | 保留 7 天已发布事件 |

### 测试覆盖

- ✅ 业务数据和 Outbox 记录同时写入
- ✅ 多个事件批量写入
- ✅ 事件唯一 ID 生成
- ✅ Worker 正确发布未发布事件
- ✅ 已发布事件不再重复发布
- ✅ 失败时重试计数增加
- ✅ 超过 maxRetries 后跳过
- ✅ cleanup 删除过期事件
- ✅ 消费者去重
- ✅ 集成测试完整流程

### 生产环境注意事项

1. **数据库事务**：实际使用时需要传入真实的 db pool 或 ORM transaction
2. **分布式部署**：多个 Worker 实例需要用分布式锁或 DB 行锁避免重复发布
3. **监控告警**：需要监控 outbox 表堆积数量、发布延迟、重试率
4. **Schema 迁移**：outbox 表添加字段需要向后兼容
5. **CDC 替代**：高吞吐场景可用 Debezium 监听 binlog 替代轮询
