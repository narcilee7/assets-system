# 安全事件响应

## 1. 事件响应流程

```
NIST SP 800-61 事件响应生命周期

┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  准备     │───▶│  检测与分析 │───▶│  遏制、根除 │───▶│   恢复   │───▶│  事后活动 │
│ Preparation│   │ Detection │   │ Eradication│   │ Recovery │   │ Post-   │
│           │    │  & Analysis│   │ Containment│   │          │   │ Incident│
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘

准备阶段：
├── 组建 CSIRT（计算机安全事件响应团队）
├── 制定事件响应计划
├── 建立通信渠道（加密、隔离）
├── 准备工具包（取证工具、镜像、日志访问）
└── 定期演练（Tabletop Exercise）

检测阶段：
├── 监控来源：SIEM、EDR、WAF、CloudTrail
├── 告警分级：
│   ├── P0（Critical）：数据泄露、勒索软件、生产入侵
│   ├── P1（High）：未授权访问、恶意软件
│   ├── P2（Medium）：策略违反、异常行为
│   └── P3（Low）：信息性告警
└── 初步评估：影响范围、数据类型、业务影响

遏制阶段：
├── 短期遏制：隔离受感染系统、阻断攻击 IP、禁用账户
├── 长期遏制：加固系统、修补漏洞、更新规则
└── 证据保全：内存转储、磁盘镜像、日志备份

根除阶段：
├── 清除恶意软件
├── 删除后门账户
├── 修补漏洞
└── 重置所有凭证

恢复阶段：
├── 逐步恢复服务
├── 增强监控
├── 验证系统完整性
└── 确认业务正常

事后活动：
├── 时间线重建
├── 根因分析（5 Whys）
├── 影响评估
├── 改进措施
└── 报告撰写
```

## 2. 事件分类与分级

| 级别 | 名称 | 响应时间 | 示例 |
|------|------|----------|------|
| P0 | Critical | 15 分钟 | 生产数据库被加密、客户数据泄露 |
| P1 | High | 1 小时 | 未授权访问管理后台、恶意软件检出 |
| P2 | Medium | 4 小时 | 异常登录尝试、策略违反 |
| P3 | Low | 1 工作日 | 扫描告警、信息泄露 |

## 3. 取证与证据保全

```
取证原则
├── 不要修改原始证据
├── 记录所有操作
├── 维护证据链（Chain of Custody）
└── 时间同步（NTP）

关键证据
├── 系统日志：/var/log、Windows Event Log
├── 应用日志：访问日志、审计日志
├── 网络流量：pcap、NetFlow
├── 内存：/proc/memdump、Volatility
├── 磁盘：dd 镜像、文件系统时间线
└── 云日志：CloudTrail、Activity Log、Audit Logs
```

## 4. 自动化响应（SOAR）

```
SOAR（Security Orchestration, Automation and Response）
├── 编排：连接不同安全工具
├── 自动化：自动执行响应动作
└── 响应：标准化的 playbook

Playbook 示例：可疑 IP 登录
├── 触发：SIEM 检测到异常 IP 登录
├── 动作 1：查询威胁情报（VirusTotal、AbuseIPDB）
├── 动作 2：如果是恶意的 → 自动阻断 IP
├── 动作 3：强制用户重新认证（撤销 session）
├── 动作 4：创建工单通知安全团队
└── 动作 5：记录到事件数据库
```
