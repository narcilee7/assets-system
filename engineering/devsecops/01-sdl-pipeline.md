# 安全开发生命周期（SDL）

## 1. 需求阶段

```
安全需求分析
├── 识别资产：数据分类（公开/内部/机密/高度机密）
├── 威胁建模：STRIDE、攻击树
├── 合规需求：GDPR、PCI DSS、等保
└── 安全验收标准：定义"什么是足够安全"

需求清单模板：
├── [ ] 所有 API 必须认证
├── [ ] 敏感数据必须加密存储
├── [ ] 用户密码必须符合复杂度要求
├── [ ] 必须支持 MFA
├── [ ] 必须有审计日志
├── [ ] 必须能通过安全扫描
└── [ ] RTO/RPO 要求
```

## 2. 设计阶段

```
威胁建模（Threat Modeling）
├── 工具：Microsoft Threat Modeling Tool、OWASP Threat Dragon
├── 方法：
│   ├── 1. 绘制数据流图（DFD）
│   ├── 2. 识别信任边界
│   ├── 3. 对每个元素应用 STRIDE
│   ├── 4. 评估风险（DREAD：Damage/Reproducibility/Exploitability/Affected users/Discoverability）
│   └── 5. 设计缓解措施
└── 设计评审检查清单：
    ├── 认证机制设计
    ├── 授权模型（RBAC/ABAC）
    ├── 数据流加密
    ├── 会话管理
    ├── 错误处理（不泄露敏感信息）
    └── 第三方依赖评估
```

## 3. 开发阶段

```
安全编码规范
├── 输入验证：白名单 > 黑名单
├── 输出编码：根据上下文编码（HTML/JS/CSS/URL）
├── 参数化查询：禁止字符串拼接 SQL
├── 最小权限：代码以最低权限运行
├── 错误处理：不暴露堆栈、路径、SQL
├── 加密：不使用自制算法
├── 日志：记录安全事件，不记录密码/Token
└── 依赖：使用经过审计的库

Pre-commit 检查
├── gitleaks：防止提交密钥
├── eslint-plugin-security：JS 安全规则
├── bandit：Python 安全扫描
└── gosec：Go 安全扫描
```

## 4. 测试阶段

```
安全测试金字塔

          ┌─────────┐
          │ 渗透测试 │  ← 手工，季度/年度
          │ (PenTest)│
          └────┬────┘
               │
          ┌─────────┐
          │  DAST   │  ← 运行时扫描
          │  (ZAP)  │
          └────┬────┘
               │
          ┌─────────┐
          │  SAST   │  ← 静态代码分析
          │(Semgrep)│
          └────┬────┘
               │
          ┌─────────┐
          │  SCA    │  ← 依赖漏洞扫描
          │ (Snyk)  │
          └────┬────┘
               │
          ┌─────────┐
          │ 单元测试 │  ← 安全逻辑测试
          │         │
          └─────────┘
```

## 5. 发布阶段

```
发布安全检查
├── 代码冻结前完成所有安全扫描
├── 安全漏洞修复完成（Critical/High）
├── 渗透测试通过（首次发布）
├── 安全配置审查（防火墙规则、密钥轮换）
├── 回滚计划（包括安全配置的版本控制）
└── 生产环境基线扫描
```

## 6. 运营阶段

```
安全运营
├── 监控：SIEM、IDS/IPS、WAF 日志
├── 告警：异常登录、权限变更、数据导出
├── 响应：事件响应手册、 on-call 轮换
├── 复盘：每次事件后的 5 Whys 分析
└── 改进：将教训反馈到 SDL 上游
```
