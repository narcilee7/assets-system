# Interview Asset Template

每个资产都应该沉淀成可复习、可运行、可复盘的最小闭环，而不是只保存一段答案。

## 文件结构

```text
README.md
impl.*
test.*
review.md
```

## README.md

```markdown
# 资产名称

## 目标

这道题或这个案例训练什么能力。

## 题目 / 场景

明确输入、输出、限制条件和真实业务背景。

## 核心考点

- 考点 1
- 考点 2
- 考点 3

## 边界条件

- 空输入
- 异常输入
- 大数据量
- 并发 / 重试 / 取消 / 超时

## 实现思路

说明关键步骤，而不是粘贴代码。

## 复杂度

- 时间复杂度：
- 空间复杂度：

## 面试追问

- 如果数据量扩大 100 倍怎么办？
- 如果需要并发安全怎么办？
- 如果失败后要重试或恢复怎么办？

## 工程迁移

它在真实工程中的对应场景是什么。
```

## impl.*

实现要求：

- 保持可运行。
- 优先写清楚边界处理。
- 复杂逻辑补少量解释性注释。
- 避免只为面试答案牺牲可读性。

## test.*

测试至少覆盖：

- 正常路径。
- 边界路径。
- 易错路径。
- 如涉及异步、并发或状态机，需要覆盖时序和失败路径。

## review.md

```markdown
# Review

## 我一开始容易写错什么

## 下次复习重点
```
