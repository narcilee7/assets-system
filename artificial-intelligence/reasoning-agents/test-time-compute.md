# Test-time Compute

Test-time Compute 是在推理时投入更多计算来提升答案质量的方法。它是从"更大模型"到"更多思考"的范式转变。

## 核心思想

```
传统：固定计算，靠模型大小提升能力
新范式：固定模型，靠推理时计算提升能力

类比：
- 传统：背更多公式（训练时投入）
- Test-time：考试时多检查几遍（推理时投入）
```

## 方法

### 1. 多数投票 (Majority Voting)

```
生成 N 个答案，取最常见的

效果：简单有效，适合有确定答案的任务
成本：N × 单次推理
```

### 2. 验证器 (Verifier / Reward Model)

```
1. 生成多个候选答案（采样或 beam search）
2. 用验证器评分每个答案
3. 选择得分最高的

训练：
- 验证器用正确/错误答案训练
- 或直接用 PRM (Process Reward Model) 评估步骤

代表：OpenAI 的 Verifier、DeepSeek 的推理模型
```

### 3. 树搜索 (MCTS / Beam Search)

```
蒙特卡洛树搜索：
1. 选择：选择最有潜力的节点
2. 扩展：生成下一步思考
3. 模拟：快速 rollout 到答案
4. 回溯：更新节点价值

适用：组合优化、数学证明、代码生成
```

### 4. 自我修正 (Self-Correction)

```
第一轮：生成初始答案
验证：检查答案合理性
第二轮：如果发现问题，修正答案

Prompt 示例：
"请检查以上回答是否有错误，如果有请修正。"

注意：
- 模型自我修正能力有限
- 通常需要外部反馈（编译器、计算器等）
```

### 5. 长思维链 (Long CoT)

```
OpenAI o1 的核心：

训练模型产生极长的内部思考：
- 分解问题
- 尝试不同方法
- 验证中间步骤
- 发现错误后回溯
- 最终给出答案

特点：
- 思考过程可能数千到数万 token
- 答案更准确但延迟极高
- 成本显著增加
```

## 计算-性能权衡

```
发现（Scaling Law for Inference）：

在固定模型下，增加 test-time compute 可以提升性能，
但边际递减。

最优策略（OpenAI 发现）：
- 简单问题：少采样
- 困难问题：多采样 + 验证器
- 不要对所有问题都用最大计算
```

## 成本分析

| 方法 | 计算增加 | 效果提升 | 性价比 |
|------|---------|---------|--------|
| 多数投票 (N=5) | 5× | 中等 | 高 |
| 多数投票 (N=100) | 100× | 边际递减 | 低 |
| 验证器筛选 | 10-20× | 显著 | 中 |
| MCTS | 50-1000× | 显著（特定任务） | 低-中 |
| 长 CoT (o1) | 10-100× | 显著 | 中 |

## 适用场景

| 场景 | 推荐方法 |
|------|---------|
| 数学竞赛 | 长 CoT + 验证器 |
| 代码生成 | 多次采样 + 执行验证 |
| 创意写作 | 少数采样 + 人工选择 |
| 实时对话 | 简单 CoT，控制延迟 |
| 科学推理 | 长 CoT + 工具使用 |

## 与模型大小的权衡

```
问题：用 10× 计算 vs 用 10× 大的模型？

发现：
- 简单任务：大模型更好
- 复杂任务：test-time compute 可能更有效
- 最优：中等模型 + 适当的 test-time compute

例子：
- GPT-4 + 1× 推理 ≈ GPT-3.5 + 10× 推理（某些任务）
```

## 常见误区

| 误区 | 正解 |
|------|------|
| Test-time compute 免费 | ❌ 显著增加延迟和成本 |
| 采样越多越好 | ❌ 边际递减，存在最优值 |
| 所有任务都适合 | ❌ 实时任务不适合高计算 |
| 可以替代大模型 | ⚠️ 互补关系，非替代 |
