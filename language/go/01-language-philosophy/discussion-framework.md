# 回答设计问题的通用思维框架

在技术讨论或面试中，回答 Go 设计问题可以遵循这个框架：

```text
1. 先说明设计目标：编译速度、工程规模、团队可读性
2. 给出具体对比：Go 的做法 vs X 语言的做法
3. 分析得失：Go 获得了什么、牺牲了什么
4. 工程实例：你在项目中遇到的真实场景
5. 演进视角：这个设计有没有在后续版本中调整？为什么？
```

## 示例回答

> "Go 没有异常，而是用多返回值 error。这是因为 Google 的大规模代码库中，异常导致的非局部控制流让代码难以理解和调试。Go 的做法让所有错误路径都显式出现在代码中。代价是 `if err != nil` 的重复代码，Go 1.13 通过 `%w` 和 `errors.Is` 改善了错误链的可追溯性。在我之前的项目中，这让我能精确定位错误来源，而不是在调用栈中追溯。"

这个框架的核心是：**不是背诵答案，而是展示你理解设计决策背后的权衡**。
