# Go 的「反模式」设计：故意不提供的特性

理解 Go 不仅要理解它有什么，还要理解它**故意没有**什么。

| 特性 | 为什么 Go 没有 | 替代方案 |
|------|-------------|---------|
| **类继承** | 避免脆弱的基类、深层耦合 | embedding + interface |
| **异常** | 避免非局部控制流 | 多返回值 error |
| **泛型元编程** | 避免编译期复杂度和代码膨胀 | 运行时灵活性（牺牲性能） |
| **断言** | 只在调试时有效，生产代码不应依赖 | 显式错误检查 |
| **不变性（immutability）** | 增加语法复杂度，Go 追求简洁 | 按值传递、不暴露内部指针 |
| **空安全（null safety）** | 需要 Optional 类型，增加类型系统复杂度 | 约定 + linter |
| **枚举类型** | 可以用常量+iota 模拟，不需要新语法 | `const (A = iota; B; C)` |
| **三元运算符** | 条件表达式可能被滥用，降低可读性 | `if-else` |
| **隐式数字转换** | 避免精度丢失的隐蔽 bug | 显式 `int64(x)` |
| **方法重载** | 同名不同签名的方法增加复杂度 | 不同方法名 |
| **迭代器协议** | 可以用 channel 或函数实现 | `for range`、闭包迭代器 |
| **finally 块** | defer 覆盖了 finally 的绝大部分场景 | `defer` |

> 这些不是"尚未实现"，而是"经过讨论后决定不提供"。
