# Review

## 我一开始容易写错什么

1. **`null` 被当成对象**：`typeof null === 'object'`，所以 `isObject` 必须显式排除 `null`。
2. **数组也被当成对象**：`typeof [] === 'object'`，需要 `Array.isArray` 排除。
3. **断言函数不返回值**：一开始写成了 `return asserts x is T`，语法错误。
4. **可选属性的校验**：`email?: string` 在守卫中要用 `'email' in x` 判断存在性，不能假设一定有该 key。

## 这个实现为什么成立

- 每个守卫只做单一职责的类型检查，组合后形成完整的业务对象校验。
- 断言函数在调用点直接收窄类型，后续代码不需要再写 `if (!isX) throw`。
- 使用 `unknown` 作为输入类型，强制开发者在校验后才能使用数据。

## 和标准库 / 框架实现的差距

- zod / valibot 等库提供了更强大的 schema 组合能力和错误路径报告。
- 手写守卫适合轻量场景，复杂校验应优先使用 schema 库。

## 工程里怎么取舍

- API 边界：使用 zod 等库做结构化校验，守卫函数做简单的内部类型收窄。
- 性能敏感路径：避免在热路径重复进行深度校验，可在入口层校验后透传已知类型。

## 下次复习重点

1. 手写 `isObject` 时先排除 `null` 和数组。
2. 区分 `x is T` 和 `asserts x is T` 的使用场景。
3. 思考如何把守卫函数组合成更复杂的逻辑（如 `isUnion`、`isIntersection`）。
