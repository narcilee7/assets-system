# Guard & Assert

## 目标

掌握 TypeScript 类型守卫和断言函数，将运行时检查与编译期类型收窄结合，避免 `as` 强制转换带来的安全隐患。

## 题目 / 场景

后端返回的数据结构是 `unknown`，需要在一系列校验后收窄为具体的业务类型，若校验失败则立即抛出可定位的错误。

## 核心考点

- `x is T`：类型守卫，返回 boolean，调用后 TypeScript 在后续代码中收窄 x 的类型。
- `asserts x is T`：断言函数，不返回值，失败时抛出错误，调用后后续代码直接按 T 处理。
- `unknown` 的渐进式收窄：优先使用 `unknown` 而非 `any`。

## 边界条件

- `null` 的 typeof 也是 `'object'`。
- 空对象 `{}` 不是任何具体业务类型的有效值。
- 数组的 typeof 是 `'object'`，需要 `Array.isArray` 区分。

## 实现思路

1. 为每个基本类型和常用结构定义守卫函数（isString、isNumber、isObject、isArray）。
2. 组合守卫函数为更复杂的业务类型守卫。
3. 使用断言函数在函数入口处快速熔断非法输入。

## 复杂度

- 时间复杂度：O(1) 单次检查，O(n) 数组遍历检查。
- 空间复杂度：O(1)。

## 面试追问

- 如果校验逻辑非常复杂，如何设计可组合的守卫函数？
- 类型守卫和断言函数在编译后的 JS 中有什么区别？
- 如何在保持类型安全的同时减少运行时校验的性能开销？

## 工程迁移

- 在 API 客户端层封装统一的 `validateResponse` 函数。
- 在单元测试中用守卫函数生成类型安全的 mock 数据验证器。
