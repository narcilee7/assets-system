# TypedEventEmitter

## 目标

实现一个类型安全的事件发布订阅系统，事件名和回调参数类型在编译期严格对应，杜绝拼写错误和参数类型不匹配。

## 题目 / 场景

在跨模块通信（如微前端、插件系统、WebSocket 消息分发）中，事件总线的事件名通常是字符串，容易因拼写错误或参数类型不一致导致运行时 bug。

## 核心考点

- 映射类型定义事件映射：`{ eventName: payloadType }`。
- `on` / `off` / `emit` 的泛型约束：确保事件名存在于映射中，回调参数类型正确。
- 内存管理：`off` 和 `once` 的正确实现，防止内存泄漏。

## 边界条件

- 重复监听同一事件：多个回调都应被执行。
- `off` 移除特定回调：需要比较函数引用。
- `emit` 时无监听者：静默返回，不抛错。
- 通配符或正则匹配：当前实现不支持，保持严格类型安全。

## 实现思路

1. 用 `Map<keyof Events, Set<Listener>>` 存储监听器。
2. `on<K extends keyof Events>` 约束事件名必须在映射中。
3. `emit<K extends keyof Events>` 约束参数类型为 `Events[K]`。
4. `off` 通过函数引用精确移除。

## 复杂度

- 时间复杂度：emit 为 O(m)，m 为该事件的监听器数量。
- 空间复杂度：O(n)，n 为总监听器数量。

## 面试追问

- 如果需要支持通配符（如 `on('*', handler)`），类型系统如何设计？
- 如果事件数量非常大，Map 的内存占用如何优化？
- 如何在 emit 时支持异步监听器，并按顺序执行？

## 工程迁移

- 封装为 React Context 或 Vue provide/inject 的全局事件总线。
- 在 Node 服务中作为模块间解耦的轻量消息总线。
