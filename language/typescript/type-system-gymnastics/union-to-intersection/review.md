# Review

## 我一开始容易写错什么

1. **直接写 `T extends infer U ? U : never`**：这样只是提取了联合类型中的每个成员，不会合并成交叉类型。
2. **忘记 `T extends any ? ...`**：如果不先分发，直接把整个联合类型放入逆变位置，可能无法正确触发分配律。
3. **不理解为什么函数参数位置有效**：需要明确协变、逆变和双变的概念，才能记住这个 trick。

## 这个实现为什么成立

- 在 TypeScript 中，函数参数是逆变的：`(x: A) => void` 是 `(x: B) => void` 的子类型，当且仅当 `B extends A`。
- 当我们把联合类型 `T` 映射到 `(x: T) => void`，由于逆变，联合会被转换为交叉：`((x: A) => void) | ((x: B) => void)` 在逆变解释下等价于 `(x: A & B) => void`。
- 再通过 `infer U` 提取参数类型，就得到了 `A & B`。

## 和标准库 / 框架实现的差距

- type-fest 的 `UnionToIntersection` 实现与此基本一致，但会额外处理 `unknown` 和 `any` 的边界。
- 某些场景下需要 `UnionToTuple`，这是更难的问题（需要借助函数重载或递归）。

## 工程里怎么取舍

- 在需要合并多个独立接口的场景（如 HOC props）非常有用。
- 如果团队不鼓励类型体操，可以用工具类（如 `type-fest`）代替手写，降低维护门槛。

## 下次复习重点

1. 手写时先写出 `(T extends any ? (x: T) => void : never)` 这一步，再写 extends infer。
2. 用实际例子验证：把 `A | B` 代入，想象类型系统如何分配和合并。
3. 对比协变位置（返回值）和逆变位置（参数）的行为差异。
