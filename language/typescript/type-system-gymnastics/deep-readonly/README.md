# DeepReadonly

## 目标

实现一个递归的 `DeepReadonly<T>`，将对象和数组的所有层级都转为只读。

## 题目 / 场景

给定一个嵌套配置对象，确保任何层级的属性都无法被重新赋值，同时保持类型推断精确。

## 核心考点

- 递归 mapped type 的终止条件。
- 如何区分对象、数组、基本类型和函数。
- TypeScript 内置 `Readonly<T>` 只处理第一层。

## 边界条件

- 空对象 `{}`。
- 数组和元组。
- 函数属性：函数本身不应被包裹成只读对象。
- 循环引用类型（理论上无法完全在类型层处理，需运行时配合）。

## 实现思路

1. 如果 `T` 是基本类型或函数，直接返回 `T`。
2. 如果 `T` 是数组，递归处理元素类型并包裹 `readonly`。
3. 如果 `T` 是对象，用 mapped type 遍历所有 key，递归应用 `DeepReadonly`。

## 复杂度

- 类型复杂度：递归深度受 TypeScript 类型实例化深度限制（默认约 50 层）。

## 面试追问

- 如果数据量扩大 100 倍（深层嵌套），TypeScript 编译性能会受影响吗？
- 如何在 DeepReadonly 的基础上支持 DeepWritable（逆向操作）？
- 如果某个属性是 Date 或 RegExp，递归是否应该把它们的内部属性也变成 readonly？

## 工程迁移

- Redux Toolkit 的 `createSlice` 要求状态不可变，DeepReadonly 可以作为状态树的类型约束。
- 配置中心下发的配置对象，在应用层应被视为不可变。
