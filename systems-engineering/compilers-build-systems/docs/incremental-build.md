# Incremental Build Design

## 目标

理解增量构建的核心挑战：依赖追踪的正确性、非确定性输入、构建系统的自举和循环依赖处理，以及如何在复杂项目中保证增量构建的可靠性。

## 场景

- 改了头文件为什么整个项目都重编了？
- 为什么有时候 clean build 能通过，增量构建却失败？
- `__DATE__` 宏为什么破坏增量构建？
- 如何处理 C++ 的模板隐式依赖？
- 编译器升级后增量构建安全吗？

## 增量构建的正确性

### 核心问题

```
增量构建正确 = 增量构建结果 ≡ Clean 构建结果

错误场景：
  1. 应该重建但没重建（遗漏依赖）
     → 运行旧代码，行为不一致
     
  2. 不该重建但重建了（过度保守）
     → 浪费时间，但能接受

目标：不漏（Sound）的前提下，尽可能减少不必要的重建
```

### 遗漏依赖的常见原因

```
1. 隐式依赖：
   - C++ 头文件包含链（间接头文件修改）
   - 编译器内置头文件（如 stddef.h）
   - 链接时自动查找的库（-l 搜索路径变化）

2. 环境依赖：
   - 编译器版本变化
   - 环境变量（PATH、CFLAGS）
   - 系统库升级

3. 非声明依赖：
   - 构建脚本读取了未声明的文件
   - 生成代码时用了随机数或时间戳
   - 符号链接指向的文件变化

4. 动态生成依赖：
   - protoc 生成 .pb.cc 时，如果 .proto 导入其他 .proto
   - 代码生成器扫描目录生成代码
```

## 依赖追踪技术

### 文件级追踪

```
Make 方式：
  目标文件.o 依赖源文件.cpp 和头文件.h
  
  问题：头文件又包含其他头文件，需要递归追踪

GCC -MD / -MMD：
  编译时自动生成 .d 依赖文件
  
  main.o: main.cpp utils.h /usr/include/stdio.h ...
  
  优势：自动发现所有包含的头文件
  局限：不追踪编译器内置宏、环境变量
```

### 内容哈希 vs 时间戳

```
时间戳：
  优点：快（stat 系统调用）
  缺点：不可靠（touch、git checkout 改变时间戳但内容不变）

内容哈希（SHA-256 / BLAKE3）：
  优点：精确，只有内容变才算变
  缺点：慢（需要读整个文件并计算哈希）
  
  优化：
    - 缓存文件哈希（Bazel 的 CAS）
    - 使用快速哈希（xxHash）
    - 先比 mtime，mtime 没变则跳过哈希

Bazel 的策略：
  - 本地文件：mtime + size 作为快速检查，变化时计算哈希
  - 远程缓存：内容寻址，天然基于哈希
```

### 编译器注入依赖

```
C++ 编译器自动包含的头文件：
  #include <...> 搜索路径中的文件
  编译器内置定义（如 __clang__、__GNUC__）
  
追踪方法：
  1. GCC 的 -H 选项：打印所有包含的文件
  2. strace -e openat：追踪所有文件打开
  3. 编译器提供依赖输出（-MD）
  
Bazel 的 C++ 工具链：
  - 声明所有内置 include 路径
  - 编译器作为输入的一部分（版本变化触发重建）
```

## 非确定性输入

### 时间戳和随机数

```
破坏增量构建的代码模式：

1. __DATE__ / __TIME__
   const char* build_time = __DATE__ " " __TIME__;
   → 每次编译结果不同，无法缓存

2. 随机数生成器初始化
   srand(time(NULL));
   → 运行时行为不同，但编译时无法检测

3. UUID 生成
   static const char* id = generate_uuid();
   → 每次编译生成不同代码

解决：
  - 用稳定值替代（如 git commit hash）
  - 集中到一个 version.cc 文件，只有它经常变
  - 构建系统提供 --stamp 选项控制是否嵌入时间
```

### 文件系统顺序

```
问题：
  readdir() 返回的文件顺序不保证
  如果构建脚本遍历目录生成代码，顺序不同 → 输出不同

解决：
  - 遍历后显式排序
  - 使用稳定的数据结构（Sorted Map）
  - Bazel 的 glob() 保证结果排序
```

## 语言特定挑战

### C++ 模板

```
问题：
  template <typename T>
  void foo(T x) { ... }
  
  foo(42);      // 实例化 foo<int>
  foo("hello"); // 实例化 foo<const char*>
  
  模板实例化可能发生在使用处（而非定义处）
  → 修改模板定义，所有使用处都需要重编

隐式依赖：
  - 模板体的变化影响所有实例化点
  - 构建系统难以静态分析模板依赖

缓解：
  - 显式实例化（explicit instantiation）
  - 预编译头（PCH）减少重复解析
  - 模块（C++20 Modules）替代头文件
```

### Go 构建缓存

```
Go 的增量构建设计：

1. 包级缓存：
   - 每个包的编译结果缓存
   - key = hash(源文件 + 导入包的 hash + 编译器版本)

2. 自动追踪：
   - 编译器自动记录所有导入依赖
   - 无需手动维护 Makefile

3. 构建约束：
   - //go:build 标签参与缓存 key
   - 不同 OS/Arch 的缓存隔离

4. 测试缓存：
   - go test 缓存测试结果
   - 只有代码或输入变化时才重新运行
```

### JavaScript/TypeScript 构建

```
问题：
  - 类型检查（tsc）和打包（webpack/esbuild）分离
  - 类型隐式依赖（全局 @types）
  - node_modules 庞大，依赖图复杂

工具：
  - Turborepo：基于哈希的任务调度，远程缓存
  - Nx：类似的 monorepo 构建系统
  - Rspack/SWC：Rust 重写，提升编译速度

缓存策略：
  - 输入：源文件 + package.json + tsconfig.json + 工具版本
  - 输出：dist/、.next/、类型声明
  - 远程缓存让 CI 和本地共享构建结果
```

## 构建系统的自举

### 循环依赖

```
场景：构建系统本身需要构建

  Bazel 源码 ──► 用旧版 Bazel 编译 ──► 新版 Bazel
  
  如果新版 Bazel 的构建规则变化：
    → 需要确保旧版能正确理解新规则
    → 或分阶段升级（bootstrap）

Go 的自举：
  1. 用 C 写 Go 1.4 编译器
  2. 用 Go 1.4 编译 Go 1.5
  3. 用 Go 1.5 编译 Go 1.6
  4. ...
  5. 用 Go N-1 编译 Go N
```

### 确定性构建（Reproducible Builds）

```
目标：相同输入 → 完全相同的二进制

要求：
  1. 源码完全相同（git commit）
  2. 构建环境相同（工具链版本、OS）
  3. 构建命令相同
  4. 无非确定性输入（时间戳、随机数、路径）

验证：
  - 两个不同机器构建
  - 比较二进制 hash
  - Debian、Tor 已实现

意义：
  - 验证二进制来自声称的源码（供应链安全）
  - 调试时符号和源码精确对应
  - 缓存有效性（内容寻址）
```

## 核心追问

1. **为什么改了注释也会导致 C++ 重编？** 头文件中的注释会被预处理包含进所有引用该头文件的源文件；即使注释不影响语义，构建系统按文件粒度追踪，头文件 mtime 变化 → 所有 include 它的 .cpp 重编
2. **Bazel 的沙箱怎么解决隐式依赖问题？** 沙箱限制编译器只能读取声明的输入文件；如果编译器试图读取未声明的头文件，open() 返回 ENOENT，编译报错，强制开发者补全依赖
3. **PCH（预编译头）对增量构建是福是祸？** 福：减少重复解析，加速编译；祸：PCH 本身变化（如修改了预编译的头文件）会导致大量文件重编；建议只把极稳定的库（如 STL）放入 PCH
4. **C++20 Modules 如何解决头文件问题？** Modules 用编译后的二进制模块接口（BMI）替代文本头文件；没有文本包含，没有重复解析；修改模块实现不影响接口时，依赖方无需重编
5. **Turborepo 的 pipeline 怎么设计才能最大化缓存命中？** 把任务拆到最细粒度（按包而非全局）、确保任务无副作用（不修改输入）、用环境变量控制而非文件修改（如 NODE_ENV 通过 turbo.json env 声明）
