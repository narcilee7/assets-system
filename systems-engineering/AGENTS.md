# Systems Engineering — Agent Guidelines

## 代码示例语言规范

1. **单一语言原则**：每个 `impl/` 实验或文档中的可运行代码块，**只使用一种编程语言实现**。不要为同一个实验提供多语言版本。
2. **语言选择优先级**：按场景选择最合适的语言，遵循以下优先级：
   - **Python**：首选。适合快速原型、脚本、数据分析、与 Linux /proc 交互、Docker 编排。
   - **Go**：适合系统编程、网络/并发原语、性能工具、需要静态编译的场景。
   - **Java**：适合 JVM 生态相关主题（GC、JFR、Async-Profiler）。
   - **TypeScript / Node.js**：适合前端/协议相关示例（WebSocket、HTTP、libuv）。
3. **避免 C/C++**：本仓库面向的读者主要熟悉 Python/Go/Java/TS，除非讨论 Linux 内核源码（L2 参考层），否则不在 `impl/` 或文档示例中新增 C/C++ 可运行代码。
4. **保留现有代码**：若目录中已存在多语言历史文件，可以保留，但后续深化同一主题时只维护/补充**一种**语言。
5. **环境兼容性**：实验脚本必须附带运行说明；Linux-only 的实验（如 epoll、/proc、tc）要在 README 中注明，并给出 Docker 运行方式。

## 文档模板（必读）

每篇主题文档按以下顺序组织：

1. 目标（L1）
2. 场景（L1）
3. 核心概念 / 原理（L1）
4. L2 深挖：源码锚定、数字锚定、边界陷阱
5. L3 可运行实验：`impl/<topic>_lab/` + README.md
6. 核心追问（L1/L2）
7. 状态表（标注深度 L1/L2/L3）

## 深度标注规范

在文档末尾的状态表中，明确标注每篇资产的深度：

```markdown
| 资产 | 深度 | 状态 |
|---|---|---|
| xxx | L2+L3 | done |
```
