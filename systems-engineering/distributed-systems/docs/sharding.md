# Sharding and Rebalance Playbook

## 目标

理解数据库和数据存储的分片策略、热点处理、迁移方案。

## 场景

- 数据量太大，单机放不下，如何分片？
- 分片键选错了导致热点怎么迁移？
- 扩缩容时如何避免服务中断？
- 跨机房分片的延迟问题怎么处理？

## 分片策略

### 哈希分片（Hash Sharding）

```
shard_key = hash(user_id) % num_shards

优点：数据分布均匀
缺点：
  - 无法范围查询（同一用户的连续数据可能在不同分片）
  - 扩容困难（需要 rehash，所有数据重新分布）
```

**一致性哈希（Consistent Hashing）**：
```
                    v-node 1
                  /
node1 --------- ring
                  \
                  v-node 2
                       \
                        v-node 3
                          \
                           node2

新增节点：只影响相邻的 v-node
数据迁移：只有 Ring 上相邻的数据需要迁移
```

### 范围分片（Range Sharding）

```
按时间或 ID 范围分片：
  shard_0: user_id 0 - 999999
  shard_1: user_id 1000000 - 1999999
  shard_2: user_id 2000000 - 2999999

优点：支持范围查询
缺点：容易产生热点（新建账户集中在 newest shard）
```

### 查表法（Lookup Table）

```
shard_id = lookup_table[shard_key]

维护一张映射表：
  user_001 -> shard_0
  user_002 -> shard_1

优点：灵活，可手动调整
缺点：
  - 额外查询开销
  - 映射表本身是单点（需要高可用）
```

## 分片键选择

### 常见分片键

| 场景 | 分片键 | 说明 |
|---|---|---|
| 用户数据 | user_id | 用户维度的查询均匀 |
| 订单数据 | user_id 或 order_id | 按用户查或按订单查 |
| 日志数据 | timestamp | 按时间范围查询 |
| 商品数据 | category_id | 按类目查询，但可能热点 |

### 热点问题

```
场景：热门商品（shard_key = product_id）

问题：
  - 热门商品的所有请求打到同一分片
  - 该分片成为瓶颈

解决：
  - 读副本：热门数据复制到多个分片
  - 缓存：Redis 缓存热门数据
  - 拆分：热点字段分离（详情、库存分开）
```

### 跨分片查询

```
问题：查询条件涉及多个分片

场景：查"用户 A 和用户 B 的所有订单"
  - 订单按 user_id 分片
  - 需要查 shard_0（用户A）和 shard_1（用户B）

方案：
  1. scatter-gather：并发查询所有分片，合并结果
  2. 副本索引：维护全局索引表
  3. 反范式：冗余数据到同一分片
```

## 迁移和扩缩容

### 迁移方案

**方案 1：双写**
```
写请求同时发到 old_shard 和 new_shard
后台任务同步历史数据
数据验证一致后切换读

问题：双写复杂度高，数据可能不一致
```

**方案 2：影子分片**
```
1. 创建 new_shard（空）
2. 写请求同时写 old 和 new
3. 增量同步：消费 binlog/change log
4. 校验：对比 old 和 new 数据
5. 切换：读 new，停止写 old
6. 清理：删除 old 数据

工具：gh-ost, pt-online-schema-change
```

**方案 3：一致性哈希重分布**
```
假设从 4 分片扩到 6 分片：
旧：hash % 4
新：hash % 6

只需迁移：4->6, 0->2, 1->3 这几个区间的数据
约 50% 数据需要迁移（而非 100%）
```

### 迁移步骤

```
阶段 1：准备
  - 评估数据量和增长趋势
  - 选择分片数（预留 50% 余量）
  - 设计分片键

阶段 2：双写
  - 同时写 old 和 new
  - 记录 binlog 位置

阶段 3：历史数据迁移
  - 使用工具（gh-ost）迁移历史数据
  - 监控迁移进度

阶段 4：数据校验
  - 对比 old 和 new 数据量
  - 抽样校验数据一致性

阶段 5：切读
  - 读切换到 new
  - 监控错误率

阶段 6：切写
  - 写切换到 new
  - 停止双写

阶段 7：清理
  - 删除 old 数据
  - 释放资源
```

## 核心追问

1. **什么时候需要分片？** 单机 QPS > 10k 或存储 > 1TB 且无法通过只读副本分流
2. **分片键选错了怎么办？** 热点：用副本或缓存；范围查询慢：用副本索引；无法迁移：考虑应用层聚合
3. **扩缩容时如何避免服务中断？** 影子分片 + 双写 + 增量同步 + 灰度切读
4. **分片后如何做全量查询？** scatter-gather 查所有分片，或维护全局索引
5. **分片和副本的区别？** 分片：数据水平拆分，每片存不同数据；副本：同一份数据多个副本，用于读写分离和容灾

## 工程迁移

- **在线扩容**：用 Vitess / TiDB 等原生支持水平扩展的数据库
- **静态分片**：提前规划分片数，预留余量，避免频繁迁移
- **热点处理**：读副本 + 缓存 + 数据分离
