# Security Baseline Checklist

## 目标

建立系统级安全基线的全面检查清单，覆盖身份认证、访问控制、网络安全、日志审计和数据保护，作为生产环境的安全准入标准。

## 场景

- 新服务上线前需要过哪些安全检查？
- 最小权限原则在实践中怎么落地？
- 如何防止 lateral movement（横向移动）？
- 安全日志要留多久，审计谁来看？
- 合规（等保/SOC2/ISO27001）对基线的要求是什么？

## 身份与访问管理（IAM）

### 身份认证

| # | 检查项 | 通过标准 | 验证方法 |
|---|---|---|---|
| 1.1 | 所有人类用户强制 MFA | SSO + TOTP/Hardware key，禁用纯密码 | 登录测试 |
| 1.2 | 服务账号禁用密码登录 | 只用证书 / IAM Role / Service Account Token | 账号审计 |
| 1.3 | 密码策略合规 | 长度 ≥ 12，复杂度要求，定期轮换（90天） | 策略配置检查 |
| 1.4 | 会话超时控制 | 空闲 30 分钟自动登出，绝对过期 8-12 小时 | 会话测试 |
| 1.5 | 登录失败锁定 | 连续 5 次失败锁定 15 分钟 | 暴力破解测试 |

### 授权与权限

| # | 检查项 | 通过标准 | 验证方法 |
|---|---|---|---|
| 2.1 | 最小权限原则（PoLP） | 每个账号只有完成工作所需的最小权限 | IAM 策略审计 |
| 2.2 | 职责分离（SoD） | 关键操作需多人审批（如发布、改配置） | 流程检查 |
| 2.3 | 定期权限复核 | 每季度审查一次账号权限，清理离职/调岗人员 | 审计报告 |
| 2.4 | 特权账号管控 | root/Admin 账号使用堡垒机，操作录屏 | 堡垒机日志 |
| 2.5 | 临时权限机制 |  elevated 权限需申请、限时、自动回收 | 工单系统检查 |

### RBAC/ABAC 模型

```
RBAC（Role-Based Access Control）：
  User → Role → Permission
  例：工程师 → DBAdmin → [read_db, write_db]

ABAC（Attribute-Based Access Control）：
  User + Resource + Environment + Action → Decision
  例：
    user.department == "finance"
    AND resource.sensitivity == "high"
    AND time.hour BETWEEN 9 AND 18
    AND source.ip IN company_network
    → ALLOW

推荐：
  - 基础权限用 RBAC（简单、可维护）
  - 细粒度控制用 ABAC（动态、上下文感知）
```

## 网络安全

### 边界防护

| # | 检查项 | 通过标准 | 验证方法 |
|---|---|---|---|
| 3.1 | 默认拒绝（Default Deny） | 防火墙规则未明确允许则拒绝 | 端口扫描 |
| 3.2 | 管理面隔离 | 运维端口（22/3389）不暴露在公网，走 VPN/堡垒机 | 网络扫描 |
| 3.3 | 微服务东西向流量管控 | 服务间默认不可通信，白名单开放 | 服务网格策略审计 |
| 3.4 | DDoS 防护 | 入口有清洗能力，SYN flood/UDP reflection 可防护 | 渗透测试 |
| 3.5 | WAF 覆盖 | 公网入口部署 WAF，规则集定期更新 | WAF 配置检查 |

### 传输安全

| # | 检查项 | 通过标准 | 验证方法 |
|---|---|---|---|
| 4.1 | 全站 HTTPS | 公网服务强制 TLS 1.2+，HSTS 启用 | SSL Labs 扫描 |
| 4.2 | 证书管理自动化 | Let's Encrypt/ACME 或内部 CA 自动续期 | 证书过期检查 |
| 4.3 | mTLS（服务间） | 内部服务双向 TLS 认证 | 抓包验证 |
| 4.4 | 弱密码套件禁用 | 禁用 RC4、DES、MD5、SHA1 | SSL Labs / testssl.sh |
| 4.5 | 证书固定（可选） | 关键 App 启用 Certificate Pinning | 客户端检查 |

### 网络分段

```
传统分段：VLAN + 子网
  - DMZ：公网入口（Nginx/WAF）
  - App：应用服务
  - DB：数据库
  - Mgmt：运维管理（堡垒机、监控）

云原生分段：安全组 + NetworkPolicy
  - Namespace 级别隔离
  - Pod 级别微分段
  - 基于标签的防火墙规则

零信任演进：
  - 不再信任任何网络位置
  - 每个访问请求都需认证和授权
  - 见 zero-trust.md
```

## 数据安全

### 分类与分级

```
数据分类：
  - 公开（Public）：公司官网信息
  - 内部（Internal）：员工手册、内部文档
  - 机密（Confidential）：客户数据、商业计划
  - 高度机密（Restricted）：源代码、密钥、个人隐私数据

分级管控：
  - 标签化：数据打上分类标签
  - 访问控制：基于标签的自动策略
  - 传输加密：Confidential 以上必须加密
  - 存储加密：Restricted 必须加密
```

### 加密与密钥

| # | 检查项 | 通过标准 | 验证方法 |
|---|---|---|---|
| 7.1 | 静态数据加密 | 数据库、对象存储、磁盘加密（AES-256） | 配置审计 |
| 7.2 | 传输中加密 | TLS 1.2+，内部服务 mTLS | 抓包验证 |
| 7.3 | 密钥管理（KMS/HSM） | 主密钥在 HSM 中，不可导出 | KMS 配置 |
| 7.4 | 密钥轮换 | 数据加密密钥（DEK）定期轮换 | 轮换策略 |
| 7.5 | 密钥访问审计 | 所有 KMS 调用记录日志 | CloudTrail/审计日志 |
| 7.6 | 敏感数据脱敏 | 日志/监控中不暴露密码、Token、信用卡号 | 日志扫描 |

## 日志与审计

### 日志要求

| # | 检查项 | 通过标准 |
|---|---|---|
| 8.1 | 全链路审计日志 | 认证、授权、数据访问、管理操作都有记录 |
| 8.2 | 日志不可篡改 | 集中存储，WORM（一次写入多次读取） |
| 8.3 | 日志保留期 | 安全日志 ≥ 180 天，合规要求 ≥ 1 年 |
| 8.4 | 日志包含上下文 | 时间戳、用户 ID、IP、操作、结果、资源 ID |
| 8.5 | 异常行为检测 | 暴力破解、异常时间登录、大量数据导出告警 |

### SIEM 集成

```
日志收集 → 归一化 → 关联分析 → 告警 → 响应

关键规则：
  - 同一账号 1 分钟内登录失败 > 5 次
  - 特权账号在非工作时间登录
  - 数据库敏感表批量导出
  - 新创建的账号立即获得 Admin 权限
  - 外部 IP 访问管理面
```

## 核心追问

1. **最小权限和职责分离冲突时怎么选？** 最小权限是常态，职责分离是特权操作的约束；关键操作（如发布到生产）必须 SoD，日常开发最小权限
2. **内部威胁怎么防？** 权限复核、行为分析（UEBA）、数据防泄漏（DLP）、敏感操作多人审批、离职即时冻结
3. **日志留 180 天成本很高怎么办？** 热日志（7 天）存 SSD/高性能存储，温日志（90 天）存对象存储，冷日志（1-7 年）存归档存储；按需检索
4. **容器里 root 和非 root 的本质区别？** root 容器突破后就是宿主机 root（如果没有 user namespace）；非 root 即使逃逸也是普通用户，攻击面大幅降低
5. **零信任和 VPN 的边界在哪里？** VPN 是"进网络就信任"；零信任是"无论在哪都要认证授权"；VPN 可以作为零信任的一个接入方式，但不是信任依据
